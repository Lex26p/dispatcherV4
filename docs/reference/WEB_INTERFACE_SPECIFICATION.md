# Dispatcher — спецификация веб-интерфейса

**Статус:** целевая продуктовая UX/UI-спецификация, синхронизированная с текущим HTML/CSS/JS-прототипом  
**Дата:** 15 июля 2026 года  
**Исходный документ:** `outputs/product_concept_dispatching_platform.md`  
**Связанный документ:** `WEB_BACKEND_API_REQUIREMENTS.md`  
**Область:** информационная архитектура, компоновка, экраны, состояния и пользовательские взаимодействия  

> **Главный принцип.** Интерфейс должен оставлять максимум места рабочим данным. Постоянная оболочка ограничивается одним компактным верхним header и сворачиваемой левой навигацией. Дополнительный контекст и краткие контекстные параметры открываются в боковых панелях. Высота экрана не расходуется на несколько уровней постоянных заголовков.

## 1. Назначение и границы документа

Документ преобразует продуктовую концепцию Dispatcher в однозначную спецификацию веб-интерфейса и является нормативным ориентиром для развития существующего тестового прототипа на простом HTML, CSS и JavaScript.

Текущий интерфейс рассчитан на одну организацию с большим количеством локаций. Переключатель организаций и multi-tenant UX в эту спецификацию не входят.

Документ определяет:

- общую оболочку приложения;
- навигацию, маршруты и перенос контекста;
- правила использования рабочей области;
- профили отображения и адаптацию к viewport;
- общие компоненты и состояния;
- интерфейсы отдельных сервисов;
- режимы дашбордов, мнемосхем, kiosk и wallboard;
- безопасное включение управления;
- минимальный набор экранов текущего прототипа;
- критерии проверки интерфейса.

Документ не определяет:

- программную архитектуру, API и протоколы;
- структуру баз данных;
- реализацию realtime, авторизации и аудита;
- окончательный фирменный стиль и графические ассеты;
- исходный код приложения;
- состав релизов и очередность промышленной разработки.

Слова **должен**, **не должен**, **следует** и **допускается** описывают целевое поведение интерфейса. Числовые размеры ниже являются проектными значениями по умолчанию, а не ограничением поддерживаемого разрешения.

## 4. Принципы простого интерфейса

### 4.1. Один основной путь

Для типовой операции должен существовать один очевидный основной путь. Альтернативный путь допускается только для ускорения опытного пользователя и не должен быть единственным способом выполнить действие.

Примеры:

- клик по строке открывает карточку справа;
- кнопка «Открыть полностью» переводит на отдельную страницу;
- кнопка «Фильтры» открывает одну общую панель;
- `Ctrl+K` открывает поиск и быстрые переходы;
- технологическая команда не запускается через скрытый жест или командную строку.

### 4.2. Progressive disclosure

Информация раскрывается последовательно:

`сводный индикатор → строка или объект на схеме → правая карточка → полная страница`.

Редкие параметры располагаются в секции **«Дополнительно»**. Технические адреса, сырые теги, выражения и привязки не показываются диспетчеру без инженерного режима.

### 4.3. Стабильность рабочего контекста

- realtime-обновление не сбрасывает фильтр, выбор и прокрутку;
- таблица не меняет порядок строк под курсором без явного действия пользователя;
- карточка не закрывается из-за обновления объекта;
- возврат из полной карточки восстанавливает список, фильтр и позицию;
- смена глобальной локации предупреждает о несохранённых изменениях;
- активный контекст всегда виден текстом, а не только подсветкой в дереве.

### 4.4. Ограниченная персонализация

Пользователь может настраивать плотность, тему, порядок колонок, сохранённые фильтры, избранное и разрешённые элементы Рабочего стола. Он не может произвольно менять смысл цветов, жизненные циклы, обязательные поля и расположение критических команд. Композиция «Моей страницы» едина для сотрудников; меняются только разрешённые данные и их effective visibility через сервис «Настройки профиля».

## 5. Профили отображения и viewport

### 5.1. Поддерживаемые профили

| Профиль | Назначение | Поведение оболочки |
|---|---|---|
| `Operator` | Текущий основной профиль `1920×1080` | Компактная плотность, быстрые клавиши, допустимое закрепление правой панели |
| `Desktop` | Перспективный обычный компьютер или ноутбук | Header, сворачиваемая навигация и адаптивные страницы |
| `Touch` | Перспективный планшет или сенсорная панель | Крупные зоны нажатия, отсутствие обязательного hover, упрощённые toolbar |
| `Kiosk` | Текущий фиксированный логический профиль интерактивной панели `1920×1080` | Общая оболочка скрыта; доступна только назначенная навигация дашборда |
| `Wallboard` | Перспективный информационный экран | Read-only, рабочий полноэкранный режим, плейлист и автоматическая смена экранов |

`1920×1080` используется как основной viewport проектирования, прототипирования и приёмки профилей `Operator` и текущего фиксированного `Kiosk`. Уменьшенные desktop viewport, touch и полноценный wallboard не являются обязательными объектами текущей реализации или проверки. Kiosk проверяет только отдельную оболочку, назначение, безопасность и основные состояния; общую адаптивность он не вводит.

### 5.2. Настройка профиля дашборда

Профиль содержит ограниченный набор понятных параметров:

- назначение: desktop, operator, touch, kiosk или wallboard;
- целевой viewport или соотношение сторон для предварительного просмотра;
- режим оболочки: обычный, компактный или скрытый;
- плотность элементов;
- минимальный размер сенсорной зоны;
- стартовый экран;
- режим масштабирования мнемосхемы;
- допустимый диапазон zoom;
- поведение при бездействии и потере связи.

Параметры наследуются от профиля организации. Дашборд переопределяет только действительно отличающиеся значения. Для назначенного терминала его профиль может дополнительно ужесточить режим до `Touch`, `Kiosk` или `Wallboard`, задать политику команд, бездействия, offline и правила воспроизведения. Пользовательские настройки применяются только в разрешённых пределах; состав плейлиста относится к отдельному content assignment.

Порядок приоритета: **ограничения безопасности терминала → профиль дашборда → пользовательские предпочтения**. Пользователь runtime не выбирает разрешение вручную. Владение прокруткой не является свободной настройкой: оно определяется типом компоновки из § 12.3.

Для `Wallboard` effective command policy всегда равна `deny`; профиль терминала не может включить на нём управляющие элементы.

### 5.3. Breakpoints — перспективное развитие

При будущем развитии адаптация должна выполняться по фактической ширине рабочей области, а не по названию устройства.

- широкая область: разрешено закрепить контекстную панель рядом с контентом;
- средняя область: левая навигация работает как rail, правая панель открывается поверх;
- узкая область: глобальный поиск и вторичные индикаторы сворачиваются в иконки, таблицы переходят к приоритетным колонкам или карточкам;
- сложные редакторы на узкой области доступны только для просмотра и предварительной проверки, если полноценное редактирование становится небезопасным.

Ориентиры будущего адаптивного этапа: область считается широкой, если после закрепления панелей центральному контенту остаётся не менее `960 px`; пороги pinning для обычной страницы и редактора уточняются отдельной адаптивной итерацией. Эти значения и поддержка узких viewport не являются критериями текущего прототипа.

На основном профиле `1920×1080` горизонтальная прокрутка всей страницы не допускается. Исключения: техническая широкая таблица с явной горизонтальной прокруткой внутри таблицы и фиксированный редакторский холст.

## 6. Глобальная оболочка

### 6.1. Геометрия по умолчанию

| Элемент | Обычный режим | Touch | Правило |
|---|---:|---:|---|
| Глобальный header | 48 px | 56 px | Единственный постоянный верхний уровень |
| Левый rail | 56 px | 64 px | Может раскрыться до 224–240 px |
| Локальная toolbar | до 40 px | до 48 px | Только на экранах, которым нужны действия или фильтры |
| Правая панель | 360–480 px | до 90% viewport | Закрываемая; на широком экране может быть изменяемой по размеру |
| Постоянный footer | отсутствует | отсутствует | Статусы размещаются в header, карточке или временном сообщении |

В обычном режиме высота рабочей области равна высоте viewport минус один global header и, только при необходимости, одну локальную toolbar. Локальная toolbar прокручивается или остаётся sticky внутри сервиса и не превращается во второй global header. В рабочем полноэкранном режиме global header скрыт; у дашборда и мнемосхемы локальная toolbar также может полностью скрываться.

### 6.2. Состав header

Слева направо:

1. Кнопка меню и знак `D`.
2. Выбранная локация или область объектов.
3. Название текущего сервиса либо компактный путь внутри дашборда.
4. Глобальный поиск.
5. Индикатор связи и достоверности платформы.
6. Счётчик событий, требующих реакции, с переходом в Диспетчер событий.
7. Иконка личных веб-оповещений с unread-счётчиком.
8. Кнопка временного режима управления, если у пользователя есть command capability.
9. Аватар активного пользователя в правом верхнем углу.

Иконка личных веб-оповещений открывает компактный inbox. При выборе сообщения выполняется переход к связанному событию, дашборду, устройству или истории тега, только если объект доступен пользователю; переход к истории тега показывается только при наличии истории. Недоступная цель не раскрывается.

Аватар открывает меню в фиксированном составе: **«Моя страница»**, **«Моя работа»**, **«Настройка оповещений»**, **«Настройки профиля»**, **«Выйти»**. У «Моей работы» допускается badge незавершённых задач. Это меню является точкой доступа к личным сервисам и не заменяет их полноразмерные страницы.

Header не должен одновременно показывать длинный breadcrumb, название организации, полное название страницы и отдельный подзаголовок. Наименование организации доступно в меню и на стартовой странице; в header при необходимости используется короткое имя.

Header всегда однострочный и не переносит элементы. Перспективное поведение при нехватке ширины уточняется на адаптивном этапе; выбранная локация, критический счётчик, личный unread-счётчик, состояние управления, связь и аватар должны оставаться доступны.

На `/home` не используется отдельный приветственный banner или второй header с именем, сменой и временем актуальности. Эти сведения размещаются в основном header либо в содержательных виджетах. Личные сервисы открываются из avatar menu, а не отдельными кнопками header.

### 6.3. Левая навигация

Основные пункты:

- Главная;
- Настройки;
- Локации;
- Дашборды;
- Диспетчер событий;
- Оборудование;
- Тренды;
- ТОиР;
- Расписания;
- Отчёты и журнал;
- Конструктор.

Пункты формируются по правам. **«Настройки» в левой навигации означают системные настройки приложения**, доступны только администраторам и полностью скрыты для остальных пользователей. «Конструктор» также показывается только при соответствующем праве. «Моя страница», «Моя работа», настройка оповещений и настройки профиля открываются из avatar menu и не дублируются в основной навигации; к «Моей работе» также ведут прямые и контекстные ссылки.

В свернутом режиме отображаются иконки с tooltip и badge. Пользователь может закрепить часто используемые сервисы в верхней части rail. При раскрытии панель по умолчанию накрывает рабочую область. Закрепление широкой панели допускается только при сохранении минимальной центральной ширины из § 5.3 и хранится как пользовательская настройка; иначе панель автоматически возвращается в overlay.

### 6.4. Правая контекстная панель

Одна панель используется для:

- краткой карточки оборудования или локации;
- подробностей события;
- быстрых свойств наряда;
- фильтров списка;
- инспектора элемента редактора;
- сведений о точке тренда.

Одновременно активен один режим панели. Если доступны и фильтры, и подробности, в заголовке используются две компактные вкладки **«Подробности / Фильтры»**; на «Фильтры» показывается badge числа активных условий. Переход к карточке не уничтожает введённые фильтры, а возврат к ним выполняется одним действием. Панель имеет заголовок, явную кнопку закрытия и действие «Открыть полностью», если существует отдельная страница.

### 6.5. Рабочий полноэкранный режим

Рабочий полноэкранный режим скрывает app shell внутри текущей browser-вкладки и доступен для:

- дашборда;
- мнемосхемы;
- тренда;
- wallboard и kiosk.

В Dashboard Editor и SVG Editor общая shell по умолчанию сохраняется. Её скрытие допускается только после отдельной осознанной команды **«Полноэкранный холст»** и не является состоянием входа в редактор.

В рабочем полноэкранном операторском режиме сохраняется компактный доступ к критическим событиям, состоянию связи и выходу. Эти элементы отображаются как небольшая overlay-панель и не создают второй header.

Это не то же самое, что browser Fullscreen API. Browser fullscreen является дополнительным действием: первый `Esc` выходит из него, повторный `Esc` — из рабочего полноэкранного режима. Browser Back возвращает предыдущий экран/режим по истории URL; явная кнопка **«Выйти»** всегда видима и ведёт туда же.

### 6.6. Режимы оболочки и владение высотой

| Режим | Верхний уровень | Прокрутка |
|---|---|---|
| Обычная страница | Один global header; не более одной локальной toolbar | Основная прокрутка принадлежит странице или рабочему списку |
| Рабочий полноэкранный | Global header скрыт; компактный status overlay | `Mimic` использует zoom/pan; `Widgets` и `Combined` — одну прокрутку страницы |
| Редактор | Global header и navigation rail сохранены; внутри рабочей области одна компактная editor toolbar | Canvas управляет pan/zoom; инспекторы прокручиваются внутри себя |
| Полноэкранный холст редактора | Общая shell скрыта явной командой; остаётся editor overlay с выходом | Canvas занимает доступный viewport |
| Kiosk/Wallboard | Общая оболочка скрыта; только назначенная dashboard-навигация | Определяется компоновкой; wallboard не требует пользовательской прокрутки |

Локальная editor toolbar не дублирует функции global header. В полноэкранном холсте global shell скрывается; в каждый момент существует ровно один владелец основной вертикальной прокрутки.

## 7. Контекст, ссылки и навигация

### 7.1. Глобальный контекст

Глобальным контекстом оболочки является только локация или разрешённая область локаций. Инженерная система, оборудование и временной диапазон являются локальными параметрами конкретного сервиса и отражаются в его URL/saved view. Смена локации обновляет поддерживающие её сервисы, но не должна молча менять открытый объект, если он находится вне новой области. В таком случае пользователь получает выбор: сохранить текущий объект или перейти в новую область.

### 7.2. Прямые ссылки

Постоянный URL должен существовать для:

- локации;
- оборудования;
- дашборда и его экрана;
- сохранённого представления событий;
- конкретного события или инцидента;
- сохранённого тренда;
- объекта обслуживания, плана ППР и наряда;
- отчёта и сменного журнала;
- опубликованной сущности конструктора.

Ссылка на элемент, который обычно открывается справа, восстанавливает родительский список и открывает панель поверх него. URL не содержит секреты и не обходит проверку прав.

### 7.3. Breadcrumb

Полный breadcrumb не занимает отдельную постоянную строку. Для глубоких страниц применяется компактный путь внутри header или локальной toolbar. Если путь не помещается, показываются первый уровень, многоточие и текущий объект.

### 7.4. Возврат и сохранение состояния

Browser Back/Forward должны работать предсказуемо. Возврат восстанавливает:

- выбранное представление;
- фильтры и сортировку;
- позицию списка;
- выбранную строку;
- экран дашборда;
- временной диапазон тренда;
- масштаб мнемосхемы в пределах текущего сеанса.

## 8. Общие интерфейсные шаблоны

### 8.1. Типы рабочих экранов

В приложении используются семь базовых типов, а не уникальная компоновка для каждого сервиса:

| Тип | Применение |
|---|---|
| `Operational canvas` | Дашборд, мнемосхема, карта, тренд |
| `Working list` | События, оборудование, наряды, пользователи, отчёты |
| `Detail workspace` | Полная карточка оборудования, инцидента, плана или наряда |
| `Person page` | «Моя страница» и страница другого сотрудника с одной устойчивой композицией |
| `Planner` | ППР, расписания, смены и загрузка ресурсов |
| `Editor` | Дашборды, SVG, шаблоны, правила и публикация |
| `Settings` | Полноразмерные личные настройки профиля/оповещений и отдельно системные настройки приложения |

`Settings` занимает рабочую область и может иметь компактную локальную навигацию. Личные маршруты `/settings/*` открываются из avatar menu или прямой ссылкой; системные находятся в `/admin/settings` и показываются в левой навигации под названием «Настройки» только администраторам. Малый modal, dropdown или правая карточка не заменяют полноразмерную страницу.

### 8.2. Working list

Единый шаблон списка содержит:

- название представления и счётчик записей;
- локальный поиск;
- несколько закреплённых быстрых фильтров;
- кнопку полной панели фильтров;
- выбор сохранённого представления;
- настройку колонок, группировки и плотности;
- таблицу или компактные карточки на узком viewport;
- массовые действия только после выбора строк;
- правую карточку выбранной записи.

Фильтры, сортировка, группировка, колонки и плотность сохраняются одним представлением. Представления бывают системными, общими и личными. У каждого представления есть URL.

### 8.3. Таблицы

- Header таблицы может оставаться видимым при прокрутке, но не дублирует toolbar.
- Изменение состояния уже видимой строки обновляется на месте без потери выбора.
- Новые строки автоматически вставляются только тогда, когда пользователь находится у начала списка и не выполняет выбор, bulk action или работу с открытой карточкой.
- После прокрутки или начала работы вставка и пересортировка замораживаются; новые записи показываются индикатором «Новых: N», а критические записи сразу отражаются в глобальном счётчике.
- Важные состояния содержат иконку и текст, а не только цвет строки.
- Inline-редактирование разрешено только для безопасных частых полей: исполнитель, срок, приоритет или рабочий статус — и только при наличии права.
- Массовое действие показывает точное количество и область объектов до подтверждения.
- Пагинация допускается для архивов; realtime-очередь использует виртуализированный список или последовательную подгрузку без вложенной страницы.

### 8.4. Фильтры

Над списком остаются только 2–4 наиболее частых фильтра и chips активных условий. Полная форма открывается в правой панели. Кнопка «Сбросить» возвращает системное состояние выбранного представления, а не очищает его непредсказуемо.

### 8.5. Формы

- Поля группируются по задаче, а не по структуре хранения данных.
- Обязательные поля видны сразу.
- Валидация выполняется рядом с полем и сохраняет введённые данные.
- Секция «Дополнительно» закрыта по умолчанию.
- Опасное изменение сопровождается анализом влияния.
- Длинный процесс оформляется последовательным мастером с возможностью сохранить черновик.
- Модальное окно не используется для длинной формы.

### 8.6. Карточки и панели

Краткая панель отвечает на вопросы: что выбрано, где находится, в каком состоянии, что требует внимания и какие следующие действия доступны. Она не повторяет всю полную карточку.

Полная карточка используется, если требуется:

- несколько содержательных разделов;
- редактирование большой формы;
- история и документы;
- длительный рабочий процесс;
- отдельная постоянная ссылка.

### 8.7. Диалоги и подтверждения

Модальный диалог применяется только для выбора, подтверждения или небольшой формы. Критический диалог показывает:

- точный объект;
- текущее состояние;
- выполняемое действие;
- ожидаемый результат;
- последствия и блокировки;
- необходимость причины или повторной авторизации.

Основная кнопка называется конкретным действием: «Подтвердить 12 аварий», «Включить насос», «Опубликовать версию», а не «ОК».

### 8.8. Уведомления интерфейса

- Короткий успешный результат показывается toast без блокировки работы.
- Ошибка, требующая решения, остаётся в панели задачи или центре оповещений.
- Длительная операция получает индикатор прогресса и может продолжаться в фоне.
- Частичный успех всегда показывает успешные и неуспешные элементы отдельно.
- Одновременно не отображается длинная стопка toast; повторы группируются.

## 9. Визуальная семантика

### 9.1. Темы

Поддерживаются светлая, тёмная и системная темы. Темы меняют фон и контраст, но не смысл состояний. Основная тема прототипа — нейтральная светлая; тёмная проверяется на тех же экранах как полноценный вариант, а не инверсия в конце разработки.

Интерфейс не должен выглядеть как набор ярких разноцветных карточек. Насыщенные цвета резервируются для эксплуатационных отклонений, режима управления и предупреждений о достоверности.

### 9.2. Плотность

| Режим | Применение | Ориентир строки таблицы |
|---|---|---:|
| Компактный | Диспетчерская, большие очереди | 32–36 px |
| Стандартный | Обычная работа | 40–44 px |
| Touch | Сенсорные экраны | не менее 48 px |

Плотность не скрывает обязательные подписи и признаки качества. Пользователь выбирает режим глобально; отдельный сервис может повысить минимальную плотность для безопасного touch-действия.

### 9.3. Severity

Применяется единая нормализованная шкала независимо от исходной системы:

- критический;
- высокий;
- средний;
- низкий;
- информационный.

Источник может хранить собственный числовой приоритет, но оператор видит понятное имя. Каждый уровень имеет сочетание цвета, иконки и текстовой метки. Цвет severity не используется для обычных кнопок и декоративных акцентов.

### 9.4. Независимые признаки

Интерфейс не объединяет в один цвет или статус:

- важность события;
- active/cleared;
- acknowledged/unacknowledged;
- assigned/unassigned;
- shelved/suppressed;
- связь с устройством;
- качество конкретного значения;
- режим обслуживания.

В таблице и карточке эти признаки показываются отдельными компактными маркерами.

### 9.5. Качество данных

Канонические состояния качества, свежести и доступности: `Good`, `Uncertain`, `Bad`, `Stale`, `Offline`, `Initializing`; их точная семантика определена в § 10.5. «Частичные данные», «нет результатов» и «данные не настроены» являются состояниями состава контента из § 10.1, а не дополнительными кодами качества отдельного значения.

Последнее известное значение сохраняется, но при устаревании получает явную подпись и время. Оно не выглядит нормальным. На тренде разрыв данных не соединяется обычной линией.

### 9.6. Типографика и числа

- Используется читаемый системный sans-serif.
- Основной текст не уменьшается ради размещения большего числа панелей.
- Числовые значения используют tabular figures для стабильной ширины.
- Единица измерения визуально вторична, но всегда доступна.
- Технический идентификатор показывается дополнительно к понятному названию.
- Длинные названия сокращаются только с tooltip и возможностью копирования полного текста.

## 10. Общие состояния и realtime

### 10.1. Состояния любого экрана или виджета

Каждый компонент предусматривает:

- первоначальную загрузку;
- актуальные данные;
- частичные данные;
- пустой результат фильтра;
- отсутствие настроенных данных;
- устаревшие данные;
- потерю realtime-соединения;
- ошибку запроса;
- отсутствие права;
- объект в обслуживании;
- объект выведен из эксплуатации.

Пустое состояние объясняет причину и предлагает следующее действие. «Нет результатов по фильтру» и «данные ещё не настроены» являются разными состояниями.

### 10.2. Skeleton и обновление

Skeleton повторяет будущую форму контента и используется только при первоначальной загрузке. При фоновом обновлении существующие данные не исчезают; отображается небольшой признак обновления.

### 10.3. Realtime gap

После разрыва realtime интерфейс:

1. сохраняет последнее состояние с отметкой времени;
2. показывает заметный, но не перекрывающий работу индикатор;
3. блокирует команды, для которых нужна актуальность;
4. после восстановления получает новый snapshot;
5. сообщает, если в период разрыва могли быть пропущены изменения.

Нельзя просто убрать индикатор после reconnect, не подтвердив актуальность отображаемых данных.

### 10.4. Системное здоровье и технологическое состояние

Сбой канала платформы не маскируется под технологическую аварию здания. В header показывается состояние платформы и связи, а в Диспетчере событий такие записи имеют тип «Системное». Пользователь может перейти к отдельному экрану здоровья.

### 10.5. Качество, свежесть и доступность источника

Во всех дашбордах, мнемосхемах, трендах и карточках используется единый словарь состояний данных:

- `Good` — значение получено и пригодно для заданного отображения;
- `Uncertain` — значение доступно, но его достоверность ограничена;
- `Bad` — источник сообщил ошибку качества либо значение непригодно;
- `Stale` — возраст последнего пригодного значения превышает допустимый для данного параметра;
- `Offline` — недоступен источник или путь получения данных;
- `Initializing` — актуальный snapshot ещё не получен.

Качество, свежесть и доступность соединения являются связанными, но не взаимозаменяемыми признаками. По запросу доступны время источника, время получения и возраст значения. Один и тот же статус имеет одинаковую семантику и визуальный компонент во всех сервисах; автор дашборда не задаёт для него произвольный цвет. `Bad`, `Stale`, `Offline` и `Initializing` не выглядят как актуальная норма и учитываются preflight технологической команды.

## 11. Безопасный режим управления

### 11.1. Выключенное состояние

По умолчанию управление выключено. Командные элементы:

- остаются видимыми, если пользователь имеет право видеть функцию;
- отображаются неактивными;
- объясняют необходимость включить управление;
- не реагируют на случайный двойной клик, drag или клавишу Enter.

Пользователь без command capability не видит кнопку включения управления и не получает подсказку, раскрывающую недоступную команду.

### 11.2. Включение

Кнопка **«Включить управление»** находится в глобальном header либо в компактной overlay-панели kiosk. Диалог включения показывает:

- область действия режима;
- длительность;
- требуемую повторную авторизацию;
- запрет передачи права другим пользователям;
- предупреждение об аудите.

После включения header получает постоянный заметный индикатор, обратный отсчёт и кнопку немедленного отключения. Режим не меняет цвет всей страницы и не ухудшает читаемость аварийных состояний.

Область режима задаётся явно как текущий объект, окно или дашборд; безопасное значение по умолчанию — текущий дашборд. Переход между его окнами сохраняет режим, но смена дашборда, выход за разрешённую локацию, reload или восстановление после разрыва доверенной realtime-сессии отключают его. Новая вкладка никогда не наследует активный режим исходной вкладки. Политика может сузить область, но интерфейс никогда не расширяет её молча.

### 11.3. Выполнение команды

Типовой сценарий:

1. Выбрать оборудование или управляющий виджет.
2. Убедиться, что режим управления активен.
3. Открыть команду.
4. Увидеть объект, текущее состояние и ожидаемый результат.
5. Пройти повторную проверку прав, связи, качества и блокировок.
6. При необходимости указать причину, PIN или второго согласующего.
7. Отправить команду.
8. Наблюдать состояния «принято», «передано», «подтверждено», «выполнено», «ошибка» или «результат неизвестен».

Закрытие диалога не скрывает выполняющуюся команду. Результат доступен в контекстной карточке и центре фоновых операций.

### 11.4. Автоматическое отключение

Режим отключается по тайм-ауту, бездействию, завершению сеанса, отзыву права, потере требуемого доверия терминала или политике объекта. Продление является отдельным аудируемым действием.

### 11.5. Kiosk

Для kiosk профиль определяет один из вариантов:

- включение кнопкой без персональной идентификации для строго ограниченных команд;
- PIN сотрудника;
- карта или иной идентификатор;
- полноценная повторная авторизация;
- полный запрет управления.

Политика конкретной команды может быть строже профиля панели. Если человек не идентифицирован, интерфейс прямо сообщает, что аудит зафиксирует терминал, но не персонального исполнителя.

## 12. Дашборды: каталог и runtime

### 12.1. Каталог

Маршрут: `/dashboards`.

При первом входе в сервис, когда последний активный дашборд для пользователя ещё не сохранён, открывается каталог. После выбора дашборда пункт «Дашборды» в основной навигации возвращает к последнему доступному дашборду. В runtime всегда присутствует явное действие «Все дашборды», возвращающее в каталог.

Каталог содержит:

- поиск;
- фильтры по локации, системе, назначению и владельцу;
- переключатель «Все / Избранные / Недавние»;
- компактные карточки или список;
- название, назначение, область данных, время обновления и состояние публикации;
- badge активных аварий, если он полезен для данного дашборда.

Карточка не перегружается миниатюрными графиками. Миниатюра используется только для узнавания. Основное действие — «Открыть»; редактирование доступно через отдельное меню при наличии права.

### 12.2. URL и внутренние экраны

Рекомендуемый шаблон: `/d/{dashboard}/{screen}`. Здесь `{screen}` — URL-alias канонического `windowId` сущности `DashboardWindow`, а не отдельная модель. Контекст передаётся безопасными параметрами либо выбранным объектом сеанса. Каждое окно имеет собственную ссылку и browser history.

После успешного открытия система сохраняет для пользователя ID дашборда и последний доступный внутренний экран. Если дашборд удалён, архивирован или право просмотра отозвано, следующий переход через основную навигацию открывает каталог. Явно введённый URL имеет приоритет над сохранённым состоянием. Режим управления, диалоги, боковые панели и незавершённые действия не восстанавливаются.

Ссылки между окнами, дашбордами, объектами, событиями и историей являются типизированными и могут переносить только разрешённый несекретный контекст и временной диапазон. Целевой экран повторно проверяет права и существование объекта. Навигационная ссылка не запускает технологическую команду.

По умолчанию список экранов открывается как компактная закрываемая боковая панель внутри runtime и не создаёт ещё одну верхнюю строку. Плитки или кнопки на самом экране могут дублировать частые переходы. В `Kiosk` общая панель скрыта, и остаётся только явно назначенная навигация внутри дашборда. Постоянное дополнительное верхнее меню дашборда не используется.

### 12.3. Три типа компоновки

| Тип | Поведение |
|---|---|
| `Widgets` | Адаптивная сетка, предсказуемый порядок, перенос элементов и вертикальная прокрутка страницы |
| `Mimic` | SVG занимает доступную область; page scroll отключён, используются zoom/pan |
| `Combined` | Полноширинный ограниченный по высоте блок мнемосхемы, затем адаптивная сетка виджетов; всей страницей владеет одна вертикальная прокрутка |

Masonry не является основным режимом: визуальный порядок эксплуатационных элементов должен оставаться предсказуемым.

### 12.4. Виджетный экран

- Сетка использует `auto-fit`: базовая минимальная ширина карточки `280 px`, но тип виджета может задать больший минимум.
- Виджеты переходят на следующую строку, не создавая горизонтальный scroll страницы.
- Предусмотрены размеры `S`, `M`, `L`: `S` занимает одну доступную колонку, `M` — две, `L` — всю строку; если ширины не хватает, размер безопасно уменьшается до полной доступной ширины.
- Заголовок скрывается, если без него содержание однозначно.
- Высота по умолчанию определяется типом виджета, но редактор может выбрать несколько предусмотренных размеров.
- Обычный виджет растёт по содержимому либо показывает краткий итог и ссылку «Открыть полностью».
- Собственный scrollbar применяется только к таблице, журналу и длинному списку.
- Runtime не допускает случайного перемещения и изменения размера.

### 12.5. Мнемосхема runtime

Обязательные действия:

- увеличить;
- уменьшить;
- `100%` — нормализованный коэффициент масштаба `1.0`, заданный при публикации, без трактовки как физический размер или DPI;
- **«Вписать»** — показать всю схему;
- **«По ширине»** — вписать по горизонтали;
- **«Сбросить к авторскому виду»** — восстановить опубликованные автором масштаб и позицию;
- панорамирование перетаскиванием ЛКМ по модели карты;
- открыть слои;
- войти в рабочий полноэкранный режим.

Итоговый runtime масштабирует мнемосхему колесом относительно положения курсора. Панорамирование выполняется удержанием и перемещением ЛКМ по свободной области, как на интерактивной карте; обычный клик по объекту остаётся выбором, а не началом pan. В отдельном или полноэкранном `Mimic` wheel над viewport сразу управляет масштабом. Во встроенном `Combined` viewport сначала получает явный focus кликом; до этого wheel прокручивает страницу, после этого управляет масштабом. `Esc`, клик вне viewport или уход фокуса возвращают wheel странице. Активность viewport заметна по ненавязчивой рамке и доступному описанию. Pinch остаётся перспективным touch-взаимодействием.

Панель zoom является компактным overlay и не масштабируется вместе с SVG. В `Combined` она закреплена в границах контейнера мнемосхемы и уходит вместе с ним при прокрутке страницы; в рабочем полноэкранном режиме она закреплена относительно viewport. Текущее значение масштаба доступно, но не занимает постоянную строку. Масштаб и позиция сохраняются в текущем сеансе; «Сбросить к авторскому виду» всегда доступно. Wheel-zoom относительно курсора и pan ЛКМ являются итоговым требованием продукта, но не обязательной реализацией текущего статического HTML-прототипа.

### 12.6. Выбор объекта

Один клик по оборудованию:

- выделяет его без изменения масштаба;
- открывает правую карточку;
- показывает ключевые значения, качество, аварии и команды;
- позволяет открыть полный объект, тренд или Диспетчер событий.

Значимая команда не выполняется непосредственно первым кликом по символу.

### 12.7. Рабочий полноэкранный режим

Режим скрывает глобальную оболочку. В углу остаются:

- выход из рабочего полноэкранного режима;
- состояние связи;
- счётчик критических событий;
- индикатор управления, если оно разрешено;
- вызов компактного меню экранов.

Эти элементы автоматически уменьшают визуальную заметность при бездействии, но критический индикатор не исчезает.

### 12.8. Базовая библиотека виджетов

Библиотека ограничивается понятными семействами:

- значение и набор значений;
- состояние оборудования;
- индикатор и шкала;
- тренд;
- таблица или список;
- события;
- временная шкала;
- навигация;
- текст, документ или инструкция;
- карта, план или мнемосхема;
- команда или панель управления;
- календарь и ближайшие ППР;
- KPI.

Составной виджет сохраняется как шаблон. Новый уникальный тип не создаётся, если задачу можно решить настройкой существующего семейства. Каждый виджет обязан иметь состояния загрузки, пустых данных, stale, bad quality, offline и отсутствия права.

### 12.9. Контекст и время

Каждое окно объявляет только действительно используемые параметры из ограниченного набора: локация, инженерная система, оборудование и временной контекст. Основные параметры размещаются в одной компактной локальной toolbar; если они не помещаются в одну строку без сжатия рабочей области, вторичные параметры переносятся в закрываемую панель **«Контекст»**, а не создают дополнительную верхнюю строку.

По умолчанию эксплуатационный экран работает в режиме `Live`. Переход в `History` является отдельным действием и создаёт постоянный индикатор режима и выбранного интервала. В `History` технологические команды недоступны; текущие значения, realtime alarm state и исторические данные не смешиваются без явной маркировки. Виджет либо использует время окна, либо показывает собственный видимый override. Произвольная цепочка, в которой один виджет скрыто получает период от другого, не допускается.

При смене контекста зависимые элементы одновременно переходят в состояние обновления. Старые значения могут оставаться видимыми только с отметкой предыдущего контекста и не считаются единым актуальным snapshot с уже обновлёнными значениями. Неразрешённый параметр, отсутствие объекта, недостаток права и broken binding показываются разными состояниями.

### 12.10. Связанный выбор и faceplate

В пределах окна существует не более одного основного выбранного объекта. Явный выбор в мнемосхеме, таблице или navigator-widget обновляет связанный контекст, правую карточку, применимые виджеты, события и тренд. Первый объект списка не выбирается автоматически без видимого правила или сохранённого контекста.

Стандартный faceplate оборудования содержит:

- идентичность и путь локации;
- operational mode, источник и владельца управления;
- quality, freshness и время последнего значения;
- ключевые значения и состояние;
- активные события и аварии;
- краткий тренд;
- типизированные переходы к полной карточке, истории и Диспетчеру событий;
- разрешённые команды, отделённые от навигации и подчинённые режиму управления.

Открытый command dialog фиксирует target. Смена выбранного объекта закрывает неподтверждённый диалог либо требует явной отмены; она никогда не переназначает команду на новый объект.

### 12.11. Приоритеты состояний мнемосхемы

Process state, operational mode и owner, alarm severity и acknowledgement, quality и freshness, maintenance и command progress отображаются независимыми признаками. `Bad`, `Stale` или `Offline` помечают значение как недостоверное, но не скрывают активную аварию; maintenance также не удаляет аварийную индикацию. Локальный или ручной режим постоянно заметен рядом с объектом.

Насыщенный цвет резервируется для отклонений и не используется как единственный канал. Состояние дублируется формой, иконкой или текстом. Декоративная постоянная анимация запрещена; анимация направления или движения процесса выполняется только при актуальных пригодных данных и переходит в явно неизвестное состояние при потере качества.

## 13. Редактор дашбордов

### 13.1. Общая компоновка

Редактор открывается внутри общей shell: global header и navigation rail остаются видимыми, а пункт **«Конструктор»** активен. В рабочей области размещаются:

- одна компактная локальная editor toolbar;
- слева — вкладки «Экраны», «Виджеты», «Структура»;
- в центре — canvas;
- справа — свойства выбранного элемента;
- снизу — закрытая по умолчанию панель ошибок;
- обе боковые панели сворачиваются;
- команда **«Только холст»** скрывает внутренние панели, не скрывая общую shell;
- отдельная команда **«Полноэкранный холст»** осознанно скрывает общую shell и имеет явный выход обратно.

Не создаются отдельные постоянные toolbar для сетки, данных, прав и публикации.

### 13.2. Локальная editor toolbar

Локальная toolbar однострочная и разделена на три зоны:

- слева: возврат, имя дашборда и состояние черновика;
- в центре: undo/redo, профиль preview и zoom canvas;
- справа: preview, проверка, «Сохранить черновик», «Опубликовать» либо «На согласование».

Toolbar не переносится на вторую строку. При нехватке места подписи вторичных действий скрываются, затем редкие действия переходят в одно overflow-меню; возврат, состояние черновика и основное действие публикации остаются видимыми.

При уходе из редактора сохранённый автосохранением черновик остаётся доступным. Если имеются изменения, ещё не попавшие в черновик, переход блокируется предупреждением с выбором сохранить или остаться.

### 13.3. Создание экрана

Пользователь выбирает один из трёх понятных шаблонов:

1. Виджеты.
2. Мнемосхема.
3. Комбинированный.

После выбора система предлагает рабочие defaults. Полный набор параметров не показывается до размещения первого содержимого.

### 13.4. Работа с виджетами

- drag-and-drop из утверждённого каталога;
- предусмотренные размеры вместо настройки каждого пикселя;
- привязка к сетке;
- выравнивание;
- копирование и дублирование;
- группировка только при практической необходимости;
- выбор конкретного оборудования либо параметра контекста;
- понятные состояния «нет данных», «плохое качество», «нет права»;
- preview команд без их выполнения.

Источник виджета задаётся статическим объектом либо именованным параметром окна. Inspector показывает входы и потребителей выбранного параметра; скрытая циклическая или многоступенчатая зависимость между виджетами является ошибкой проверки. Локальный time override требует понятной причины и всегда заметен в runtime.

Применение шаблона составного виджета создаёт редактируемый snapshot. Последующее изменение шаблона не меняет существующие дашборды молча; управляемое массовое обновление требует отдельной версии и impact preview.

Мнемосхема является отдельной версионируемой сущностью. На dashboard-canvas размещается ссылка на опубликованную мнемосхему и её author view, а действие **«Редактировать мнемосхему»** открывает отдельный SVG-редактор из § 14; SVG-path и bindings не редактируются внутри редактора дашборда.

### 13.5. Инспектор

Показывает только свойства выбранного типа. Основные группы:

- содержание;
- данные;
- контекст и время;
- отображение;
- действия;
- доступность;
- responsive;
- дополнительно.

Технический режим с выражениями и диагностикой привязки открывается отдельно и не смешивается с основной формой.

### 13.6. Проверка и публикация

Проверка охватывает ссылки, привязки, права, состояния качества, команды, размеры touch-зон, выход за границы, зависимости и контраст. Клик по ошибке выделяет соответствующий элемент.

Результат проверки привязан к точной сохранённой revision черновика. Несохранённые изменения не проверяются, а любое изменение после проверки делает validation неактуальным. Публикация текущего несохранённого или непроверенного состояния запрещена. То же правило применяется к SVG Editor.

Черновик не влияет на runtime. Публикация показывает изменения и затронутые экраны, роли и терминалы. Опубликованная revision неизменяема и применяется целиком: runtime не получает комбинацию старых и новых окон, виджетов, bindings или символов. Открытый сеанс переключается на новую revision только после её полной доступности и показывает краткое уведомление, если для безопасного применения требуется обновить экран. Откат создаёт новую версию.

### 13.7. Совместная работа

Редактор показывает присутствующих пользователей и занятый ими элемент. Автосохранение защищает черновик, но не публикует его. Конфликтующие изменения сравниваются; система не выбирает победителя молча и не перезаписывает чужую работу по принципу последнего сохранения.

## 14. Редактор SVG-мнемосхем

### 14.1. Компоновка

Используется тот же встроенный в общую shell editor workspace; global header и navigation rail сохраняются, а «Конструктор» остаётся активным:

- одна toolbar;
- слева — «Символы», «Слои», «Структура»;
- центр — SVG-canvas;
- справа — «Свойства», «Привязки», «Состояния»;
- панель проверки закрыта снизу;
- режим «Только холст».

Единая оболочка уменьшает обучение и позволяет повторно использовать действия preview, версии и публикации.

### 14.2. Основные режимы

- выбор;
- примитивы и текст;
- линии/трубы/соединения;
- привязки;
- preview.

Активен один режим. Инструменты режима меняют центральную зону одной toolbar, а не добавляют новую строку. Зоны и правила overflow совпадают с § 13.2.

### 14.3. Символы

Библиотечный символ имеет версию, публичные параметры, точки соединения, именованные состояния и безопасное неизвестное состояние. Экземпляр на сцене показывает только публичные параметры; внутренняя SVG-структура раскрывается в расширенном режиме.

Обновление символа:

1. показывает список использований;
2. сравнивает версии;
3. проверяет совместимость привязок;
4. предлагает обновить выбранные экземпляры;
5. не меняет опубликованные мнемосхемы молча.

### 14.4. Canvas

Поддерживаются zoom/pan, сетка, snap, направляющие, слои, выравнивание, множественный выбор, блокировка и скрытие. Overlay-кнопки и навигационные элементы могут быть помечены как не участвующие в zoom.

SVG Editor позволяет сохранить текущий transform как default author view мнемосхемы. При размещении на конкретном dashboard screen Dashboard Editor может задать author-view override для данного профиля. Runtime-команда «Сбросить к авторскому виду» использует override размещения, а при его отсутствии — default author view; `100%` меняет только коэффициент масштаба и не подменяет author view.

Semantic zoom допускается как расширенная функция: второстепенные подписи и детали скрываются на малом масштабе. Он не обязателен для текущего статического прототипа.

### 14.5. Привязки и состояния

Редактор предлагает типовые правила вместо свободного скрипта:

- значение/текст;
- видимость;
- вариант символа;
- цвет состояния;
- положение/вращение;
- направление процесса;
- переход;
- разрешённая команда.

Для любого динамического элемента обязательно определяется поведение при неизвестном, плохом и устаревшем значении.

### 14.6. Безопасный preview

Preview позволяет имитировать норму, предупреждение, аварию, потерю связи, обслуживание и отсутствие права. Реальные команды заблокированы. Отдельный строго обозначенный режим проверки команд возможен только по политике и не включается автоматически.

### 14.7. Импорт и проверка

Импортированный SVG проходит очистку встроенных скриптов, обработчиков, опасных внешних ресурсов и неподдерживаемых конструкций. Проверка выявляет:

- отсутствующий или некорректный `viewBox`;
- повторяющиеся идентификаторы;
- элементы вне холста и нулевого размера;
- потерянные ресурсы;
- конфликтующие привязки;
- отсутствие unknown/offline-состояния;
- недоступную без цвета индикацию;
- чрезмерную анимацию;
- слишком маленькие touch-зоны;
- команды без прав и обратной связи.

Клик по ошибке переносит к конкретному элементу.

## 15. Локации, карты и планы

### 15.1. Основной экран

Маршрут: `/locations`.

Рабочая область поддерживает три режима:

- дерево/список;
- географическая карта;
- план территории, здания или этажа.

Левый browser локаций является закрываемой панелью внутри сервиса и не дублирует глобальную навигацию. Выбор синхронизируется с картой и правой карточкой.

### 15.2. План

План занимает всю доступную область и поддерживает zoom/pan. Слои открываются в правой панели либо компактном overlay. Доступны слои инженерных систем, событий, работ и пользовательских данных.

Клик открывает краткую карточку. Двойной клик и hidden gestures не являются обязательным способом навигации. Вложенная локация показывает счётчик ситуаций и позволяет перейти внутрь.

### 15.3. Карточка локации

Краткая панель содержит:

- название и путь;
- агрегированное состояние с объяснимой причиной;
- активные события и инциденты;
- недоступное оборудование;
- ближайшие работы;
- связанные дашборды и документы;
- действия «Открыть», «Открыть дашборд», «Показать события».

## 16. Оборудование

### 16.1. Реестр

Маршрут: `/equipment`.

Основной вид — рабочая таблица. Дополнительно доступно дерево по локациям и системам. Основные фильтры:

- локация;
- инженерная система;
- тип;
- состояние связи;
- эксплуатационный статус;
- наличие активных аварий;
- наличие плана ТОиР;
- ответственный.

Клик по строке открывает правую карточку. Массовые изменения выполняются только после preview перечня объектов.

### 16.2. Краткая карточка

Содержит:

- имя, тип и локацию;
- связь и последнее обновление;
- эксплуатационный режим;
- ключевые параметры с качеством;
- активные события;
- ближайший ППР;
- доступные безопасные действия;
- ссылки на полную карточку, тренд, мнемосхему и события.

### 16.3. Полная карточка

Маршрут: `/equipment/{id}`.

Одна компактная строка разделов:

- Обзор;
- Параметры;
- Тренды;
- События;
- Связи;
- Документы;
- ТОиР;
- Конфигурация — только для инженера;
- Активность.

Идентификация объекта и вкладки образуют одну компактную sticky-строку. После начала прокрутки вторичные паспортные данные сворачиваются, но имя, состояние, локация и текущий раздел остаются видимыми. Разделы используют маршруты, поэтому их можно открыть по ссылке; большая паспортная шапка не повторяется на каждой вкладке.

В инженерном режиме полная карточка также предоставляет диагностику подключения и технические сведения точки/источника: качество, source/receive/processed timestamps, частоту обновления, последнее успешное чтение и последние ошибки. Для этого не требуется отдельная постоянная верхняя вкладка.

### 16.4. Параметры

Строка параметра показывает имя, значение, единицу, качество и время. Подробности источника и диагностика открываются справа. Команды визуально отделены от измерений и не маскируются под обычное редактируемое поле.

### 16.5. Универсальный редактор устройств

Маршрут: `/equipment/add`.

Редактор не разделяется на одиночный и пакетный режимы. Его основа — одна staging-таблица: команда ручного добавления создаёт пустую строку, а загрузка файла CSV добавляет распознанные строки в эту же таблицу. Текстовое поле для вставки CSV не используется.

Выбор строки открывает справа полноразмерную форму устройства со всеми общими и протокол-специфичными полями. Ошибки структурной проверки показываются рядом с соответствующими полями. Текущий тестовый прототип ограничен **Modbus TCP** и **SNMP**; неподдерживаемый протокол оставляет строку ошибочным черновиком и не преобразуется автоматически.

Секретные значения маскируются; команда показа раскрывает только значение, введённое в текущей несохранённой форме. Сохранённый secret backend повторно не возвращает. Секреты не попадают в preview, результаты тестов или журналы интерфейса. Отдельная ссылка на хранилище секретов не является обязательным полем тестового прототипа.

### 16.6. Staging и применение

Новая строка по умолчанию получает `create`; совпадение с существующим объектом — `skip/review`. `update` доступен только после явного выбора администратора. Действие `delete` отсутствует, а отсутствие строки в CSV никогда не удаляет существующее оборудование или точку.

Применение конкретной строки блокируют только ошибки структуры данных и уникальности. Ошибочные строки остаются черновиками и не мешают применить валидные строки. Импорт не активирует конфигурацию напрямую.

### 16.7. Проверка связи и тестовый опрос

Для выбранной строки доступны независимые команды **«Проверить связь»** и **«Запустить тестовый опрос»**. Их результаты информативны и не блокируют добавление устройства. Проверка связи показывает этап, длительность и понятную ошибку; тестовый опрос — исходное имя, значение, качество, время и ошибку чтения. Повторный запуск не стирает заполненную форму.

После добавления устройство не получает искусственный эксплуатационный статус «Не проверено»: фактическое состояние определяется штатным опросом.

### 16.8. Копирование, шаблоны и параметры SNMP

Выбранное устройство можно скопировать с указанием количества экземпляров. Автоматически увеличиваться может только **Modbus Unit ID**; IP-адрес никогда не инкрементируется.

Шаблоны встроены в редактор. Пользователь может сохранить выбранное настроенное устройство как шаблон, а в picker — выбрать, просмотреть или удалить шаблон. В шаблон не входят ID, имя, IP-адрес, Modbus Unit ID и секреты. При выбранной staging-строке шаблон заполняет её; если строк нет, он создаёт новый редактируемый черновик. Шаблон никогда автоматически не изменяет уже созданные устройства.

Для **SNMP v2c** поле community по умолчанию содержит `public`, отображается маскированно и имеет команду показа; предупреждение о `public` не блокирует добавление. Настройки **SNMP v3** находятся в отдельной секции формы.

## 17. Диспетчер событий

### 17.1. Назначение и маршрут

Маршрут: `/events`.

Сервис является единым оперативным журналом и очередью реакции и не разделяется на отдельные приложения для событий и аварий. При этом UX различает:

- **событие** — неизменяемый точечный факт, обычно без acknowledgement;
- **аварийное occurrence** — экземпляр состояния с независимыми `active/cleared` и `acknowledged/unacknowledged`;
- **инцидент** — отдельный управляемый процесс реагирования, открываемый из представления этого же сервиса;
- **технологическое / системное / операторское / ТОиР** — категория или источник, а не взаимоисключающий lifecycle-тип.

Действия оператора включаются в общий список только при самостоятельной эксплуатационной ценности; полный activity/audit остаётся в административном журнале. Для неприменимого состояния колонка показывает `—`, а недопустимое действие отсутствует.

На первом открытии применяется представление по роли. Для оператора рекомендуется «Требуют реакции», но представление «Все события» всегда доступно одним действием и использует тот же список.

### 17.2. Компоновка

Экран содержит:

- одну локальную toolbar;
- таблицу во всю оставшуюся высоту;
- правую панель фильтров либо подробностей выбранной записи;
- необязательную компактную строку активных filter chips;
- отсутствие отдельной постоянной библиотеки представлений.

В toolbar размещаются:

- выбор сохранённого представления;
- режим «Realtime / История»;
- локальный поиск;
- быстрые фильтры severity и состояния;
- кнопка полной панели фильтров;
- счётчик результата;
- настройка колонок;
- меню экспорта при наличии права.

Toolbar всегда остаётся в одной строке. При нехватке ширины счётчик результата и настройка колонок переходят в overflow-меню, поиск сворачивается в кнопку, затем остаются выбор представления, `Realtime / История`, severity и фильтры. Активные chips переносятся только в отдельную необязательную строку, которую можно скрыть; сама toolbar не переносится.

### 17.3. Системные представления

- Все события;
- Требуют реакции;
- Активные аварии;
- Неподтверждённые;
- Назначенные мне;
- Возвращены к норме, но не подтверждены;
- Связаны с незакрытым инцидентом;
- Shelved и suppressed;
- В обслуживании;
- Системные;
- История.

Несколько часто используемых представлений можно закрепить как компактные переключатели. Остальные находятся в одном списке. Системное представление нельзя незаметно перезаписать персональными изменениями; пользователь сохраняет копию.

### 17.4. Таблица

Колонки по умолчанию:

1. Severity.
2. Вид записи: событие или аварийное occurrence.
3. Состояние условия.
4. Подтверждение.
5. Время возникновения.
6. Длительность.
7. Сообщение.
8. Локация.
9. Оборудование или источник.
10. Ответственный.
11. Категория/источник: технологическое, системное, операторское или ТОиР.
12. Специальные признаки: shelved, suppressed, maintenance, ошибка доставки.

На небольшом экране сохраняются severity, состояние, время, сообщение и источник; остальные данные открываются в карточке.

Состояние, acknowledgement, assignment, окончание shelving и качество уже видимой записи обновляются на месте немедленно. В idle-состоянии новые строки вставляются в позицию активной сортировки/группировки; в системном realtime-представлении по умолчанию это начало списка с сортировкой «сначала новые». После прокрутки, выбора, открытия карточки либо bulk action вставка и пересортировка замораживаются и появляется кнопка **«Новых: N»**. Критическая новая авария в любом случае сразу увеличивает глобальный счётчик. Выделение, массовый выбор и открытая карточка сохраняются.

### 17.5. Независимые состояния

- `severity` отвечает на вопрос «насколько важно»;
- `active/cleared` — существует ли условие сейчас;
- `acknowledged/unacknowledged` — увидел ли его оператор;
- `assigned/unassigned` — кто занимается;
- `shelved` — кто и до какого времени временно отложил реакцию;
- `suppressed` — какое правило или режим исключил оперативную реакцию;
- `maintenance` — связан ли сигнал с обслуживанием.

Подтверждение не назначает пользователя автоматически и не означает возврат к норме.

Представление **«Требуют реакции»** включает активные неподтверждённые аварии, cleared-occurrences с обязательным неподтверждённым acknowledgement и явно настроенные системные записи, требующие acknowledgement. Assignment само по себе не добавляет и не удаляет запись из этого представления.

### 17.6. Фильтры и сохранение

Базовые фильтры:

- период или realtime;
- severity;
- тип;
- состояние условия;
- подтверждение;
- локация;
- инженерная система;
- оборудование;
- категория;
- ответственный;
- shelved/suppressed/maintenance;
- свободный текст.

Сохранённое представление включает фильтры, сортировку, группировку, колонки, плотность и режим времени. Оно может быть личным, общим для группы или системным и имеет URL.

### 17.7. Правая карточка события

Панель содержит:

1. Severity, состояние, подтверждение и тип.
2. Сообщение, локацию, источник, значение и качество.
3. Основные разрешённые действия.
4. Краткую инструкцию оператору.
5. Хронологию возникновения, возврата, подтверждений, назначений и уведомлений.
6. Мини-тренд вокруг события при наличии истории.
7. Связанные события, аварии и вероятную причину.
8. Переходы к оборудованию, мнемосхеме, полному тренду и инциденту.

При `suppressed` карточка явно показывает правило/режим, владельца, начало, окончание или условие снятия и ссылку на обслуживание/политику. Запись остаётся доступна в системном представлении и истории; обычной операторской кнопки `Unsuppress` нет.

Подробности не разбиваются на несколько полноэкранных страниц. Для сложного реагирования создаётся или открывается отдельный инцидент. В зависимости от состояния карточка показывает не более одного primary и одного secondary action; остальные действия находятся в меню.

### 17.8. Действия

| Действие | Смысл |
|---|---|
| Подтвердить | Зафиксировать, что оператор увидел состояние |
| Назначить | Назначить ответственного или группу |
| Создать инцидент | Начать управляемое реагирование |
| Создать заявку | Передать ситуацию в ТОиР |
| Shelve | Временно отложить с обязательной причиной и временем окончания |
| Unshelve | Досрочно вернуть применимое активное occurrence в рабочую очередь |
| Открыть тренд | Исследовать параметр вокруг события |
| Открыть мнемосхему | Перейти к месту и процессу |

`Suppress` обычно является результатом правила, обслуживания или административного решения и не показывается как обычная операторская кнопка. `Disable` меняет конфигурацию источника и доступно только инженеру в отдельном сценарии.

Lifecycle-действия доступны только для актуального применимого alarm occurrence в режиме `Realtime`. `История` read-only: разрешены экспорт, открытие контекста и связанных сущностей. Если для исторической записи существует актуальное occurrence, карточка показывает **«Открыть текущее состояние»**. Cleared-occurrence с обязательным acknowledgement остаётся в актуальной очереди до подтверждения.

Комментарий при acknowledgement может быть необязательным или обязательным по классу/severity и сохраняется в timeline. Отдельное действие «Добавить комментарий» не меняет acknowledgement. Shelving не требует общего режима управления оборудованием, но требует отдельного права; политика может потребовать повторную авторизацию. При окончании срока активное occurrence автоматически возвращается в рабочую очередь и получает запись в timeline.

### 17.9. Массовые действия

После выбора строк toolbar показывает точное количество и область. Разрешаются подтверждение, назначение, создание группового инцидента и экспорт. Массовый shelving допускается только по политике.

Перед выполнением preview разделяет eligible/ineligible записи и объясняет причины. Для bulk shelving задаются единые причина и срок; результат показывает успешные и отклонённые строки и сохраняется в аудите. Preview группового инцидента показывает локации, системы, временной разброс и уже связанные инциденты, предупреждая о несвязном наборе.

Команды «Подтвердить всё» и «Shelve всё» без показанного scope запрещены. Критические записи могут требовать отдельной обработки, причины или повторной авторизации.

### 17.10. Архив и аудит

История read-only и отвечает на вопросы: кто, что, когда, где и почему сделал. Журнал доставки оповещений связан с событием, но не заменяет его жизненный цикл. Bulk action хранит общий запрос и перечень фактически затронутых и отклонённых записей.

## 18. Инциденты

### 18.1. Создание

Инцидент создаётся из события, группы событий, локации или вручную. Перед созданием система показывает похожие активные инциденты. Исходные события и snapshot контекста прикладываются автоматически.

### 18.2. Рабочая область

Список активных инцидентов открывается по `/events?view=incidents` внутри пункта навигации **«Диспетчер событий»**; отдельного второго диспетчера не создаётся. Это service-level subview со своей incident-таблицей, колонками и фильтрами, а не saved filter набора event/alarm из § 17.4. Полная карточка имеет маршрут `/incidents/{id}`.

Компоновка:

- компактная идентификационная строка и основные действия;
- центральный контекст: план, мнемосхема либо краткая сводка;
- правая рабочая панель с регламентом, участниками и задачами;
- единая временная шкала ниже или в переключаемом центральном представлении;
- отсутствие отдельного header для каждого блока.

### 18.3. Содержание

- номер, категория, приоритет и состояние;
- локация и затронутые системы;
- координатор, участники и дежурные группы;
- связанные события и оборудование;
- регламент и обязательные шаги;
- задачи разных служб;
- команды и их фактические результаты;
- комментарии и вложения;
- сроки и эскалация;
- хронология действий;
- итог и причина закрытия.

### 18.4. Жизненный цикл

`Новый → Назначен → Принят → В работе → Локализован → Нормализован → Проверка → Закрыт`.

Дополнительные состояния «Ожидает внешнего действия» и «Отменён» требуют причины. Возврат оборудования к норме не закрывает инцидент автоматически, а закрытие инцидента не скрывает активную аварию.

### 18.5. Критический режим

Для пожарного и другого критического инцидента интерфейс собирает план, источник, зоны, состояние связанных систем, регламент и подтверждённые результаты. Автоматические действия сертифицированной системы визуально отделяются от доступных операторских команд.

## 19. Оповещения

### 19.1. Разделение интерфейсов

Существуют три разных интерфейса:

- **Личный inbox веб-оповещений** — адресованные текущему пользователю сообщения внутри приложения;
- **«Мои оповещения»** — простые персональные подписки, график, отсутствие и каналы;
- **«Политики оповещений»** — административная маршрутизация, обязательные получатели, эскалации и шаблоны.

Личный inbox не заменяет Диспетчер событий: он показывает только доставленные конкретному пользователю сообщения, тогда как Диспетчер событий остаётся полным рабочим журналом доступных событий и аварий. Обычный пользователь не редактирует дерево правил и произвольные выражения.

### 19.2. Мои оповещения

Базовый маршрут: `/settings/notifications`. Это полноразмерный личный сервис, открываемый действием **«Настройка оповещений»** из avatar menu или прямой ссылкой; он не является системными настройками приложения или частью «Моей страницы». На основном профиле используется компактная локальная боковая навигация. Каждый пункт имеет устойчивую ссылку:

1. `/settings/notifications/effective` — **Что я получаю**: итоговая effective policy, включая обязательные правила read-only.
2. `/settings/notifications/subscriptions` — **Мои подписки**: дополнительные подписки.
3. `/settings/notifications/schedule` — **Расписание и отсутствие**: рабочие часы, quiet hours, отпуск и замещение.
4. `/settings/notifications/channels` — **Каналы**: адреса, состояние и тест.

На обзорном экране простым текстом показано текущее поведение: какие критические уведомления придут сейчас, какие дополнительные подписки приостановлены и кто замещает пользователя.

### 19.3. Подписка пользователя

Простая форма:

1. Где: локация, система или оборудование.
2. Что: события, аварии, инциденты, работы или отчёты.
3. Условие предметной области: severity для событий/аварий, priority/status для инцидентов и работ, schedule/frequency для отчётов.
4. Когда: всегда или по личному расписанию.
5. Куда: разрешённые каналы.

Подписка расширяет обязательный набор и не может его уменьшить. Совпадение обязательного и личного правила применяет более строгую обязательную срочность и не создаёт дублирующую доставку одному человеку по одному каналу.

### 19.4. Обязательные правила

Организация определяет обязательного получателя, срочность, допустимые или обязательные классы каналов и эскалацию. Пользователь задаёт endpoint и предпочтения только в разрешённых пределах. Правило показывается с замком, владельцем и объяснением: роль, локация, дежурство или замещение. Если изменение запрещено, интерфейс не показывает неработающий toggle; effective policy перечисляет все правила, сформировавшие итог.

При совпадении нескольких обязательных правил маршруты суммируются. Одинаковая пара recipient/channel дедуплицируется с самым ранним временем и наименее задерживающей группировкой; разрешённые quiet windows пересекаются, а остановка маршрута наступает только после выполнения всех применимых обязательных stop conditions. Если правила требуют взаимоисключающих каналов или иного результата, который нельзя слить однозначно, публикация блокируется до явного устранения конфликта. Preview показывает каждое contributing rule и полученный итог.

### 19.5. Каналы

Карточка канала содержит:

- тип;
- маскированный адрес;
- подтверждён/не подтверждён;
- включён/отключён;
- порядок или резервный статус;
- последний успешный тест;
- кнопку «Отправить тест»;
- последнюю ошибку доставки.

Канал отделён от подписки. Изменение email не требует редактировать все правила. Обязательный канал нельзя отключить без рабочего разрешённого замещения или резервного маршрута; интерфейс сразу объясняет блокировку. Публикация обязательной политики блокируется, если у обязательного получателя нет рабочего endpoint или допустимого резерва.

### 19.6. Расписание и quiet hours

Доступны presets:

- всегда;
- только рабочее время;
- пользовательский недельный график.

Всегда показываются timezone и ближайшее переключение. Quiet hours влияют только на разрешённые политикой доставки. Критическое обязательное уведомление не исчезает из-за личной настройки.

### 19.7. Отпуск и замещение

Отпуск не является простым mute. Пользователь задаёт период и заместителя либо дежурную группу. Состояния записи: `Черновик → Ожидает принятия/согласования → Активен → Завершён/Отменён`. Перед сохранением интерфейс показывает:

- какие дополнительные подписки приостановятся;
- какие обязанности требуют передачи;
- кто фактически будет дежурным;
- существует ли разрыв покрытия;
- нужно ли согласование.

Пауза только дополнительных подписок применяется сразу. Отсутствие, затрагивающее обязательную обязанность, не становится активным до принятия замещения подходящим пользователем, покрытия дежурной группой либо явного административного override с причиной. График показывает базовое расписание, overrides и итогового получателя. После возвращения пользователь получает сводку.

### 19.8. Административная политика

Маршрут: `/admin/notifications`.

Настройка состоит из линейных шагов, а не обязательного визуального графа:

1. Область и условия.
2. Первый получатель и канал.
3. Следующий получатель через заданное время, если не выполнено выбранное условие остановки.
4. Резервная группа.
5. Повтор и условие остановки.

После выбора типа объекта UI предлагает конечный набор применимых **«Остановить эскалацию при»**: для alarm occurrence — acknowledgement и, если разрешено классом, clearing; для инцидента — принятие или закрытие; для работы — принятие или выполнение. Для immutable event без acknowledgement доступна one-shot доставка либо продолжение через созданный incident/workflow. Неприменимое условие не показывается. Delivery receipt сообщения не считается acknowledgement. Сложные matchers доступны в «Дополнительно» только экспертной роли.

### 19.9. Preview effective policy

Перед публикацией администратор задаёт тестовое событие, дату и время. Preview показывает:

- совпавшие правила;
- фактических получателей;
- каналы;
- моменты эскалации;
- влияние quiet hours, отпуска и override;
- группировку;
- причины выбора и contributing rules;
- разрыв покрытия, нерабочий канал или конфликт правил.

Изменение политики версионируется. Уже начавшаяся эскалация не меняет смысл задним числом.

### 19.10. Группировка доставки

Группировка влияет на сообщения, но не объединяет исходные события в журнале. Для пользователя применяются presets:

- немедленно;
- объединить коротким окном;
- периодическая сводка.

Личная подписка не может перевести обязательную critical-доставку в digest или увеличить её окно. Административная группировка critical допускается только явной обязательной политикой и видна в preview. Продвинутые интервалы доступны администратору, но не выставляются в основной форме.

### 19.11. Личный inbox веб-оповещений

Отдельная иконка в header показывает количество непрочитанных личных сообщений и открывает компактный список. Сообщение содержит время, тип, краткий текст, признак прочтения и ссылку на источник. При наличии права пользователь может перейти к связанному событию, дашборду, устройству или истории тега; переход к истории тега доступен только когда история существует. Проверка права выполняется при каждом открытии.

Прочтение сообщения изменяет только состояние inbox и не означает acknowledgement события, принятие ответственности или закрытие аварии. Inbox допускает переход в полный личный список `/notifications`, но не предоставляет фильтры и массовые операции Диспетчера событий.

## 20. Тренды и история

### 20.1. Рабочая область

Маршрут: `/trends` или ссылка на сохранённое исследование.

График занимает всю оставшуюся высоту. Одна toolbar содержит только:

- период;
- live/pause;
- pan/zoom;
- reset;
- курсор/X-trace;
- fullscreen;
- сохранение представления.

Добавление сигналов, оси, стили, агрегаты, аннотации и экспорт находятся в правой панели.

### 20.2. Добавление параметров

Параметр добавляется:

- из активного оборудования;
- из карточки параметра;
- через поиск;
- из сохранённого набора.

Постоянное большое дерево всех технических тегов не показывается. Несовместимые единицы предупреждаются и получают отдельные оси только по явному решению.

### 20.3. Взаимодействие

- wheel и drag масштабируют/перемещают временную шкалу;
- double-click не является единственным reset;
- выбранный диапазон показывает min/max/average и качество;
- легенда закрывается или переносится вправо;
- клик по отметке события открывает карточку;
- переход из события заранее устанавливает интервал вокруг возникновения.

### 20.4. Качество и аннотации

Разрывы, плохое качество и ручные значения отображаются явно. На временной шкале доступны события, команды, обслуживание, изменение режима и публикация конфигурации. Слишком большое число аннотаций группируется по масштабу.

### 20.5. Сохранение

Представление сохраняет сигналы, период, оси, фильтры и аннотации. Оно имеет URL, может быть личным или общим и прикладывается к инциденту, заявке или отчёту.

## 21. Расписания, режимы и сценарии

### 21.1. Основные представления

Достаточно двух основных представлений:

- календарь;
- список.

Календарь показывает фактическое применение правил, исключения и ручные переопределения. Список предназначен для поиска, публикации и массового контроля. Дополнительные режимы не создаются без доказанной необходимости.

### 21.2. Редактирование расписания

Форма показывает недельный шаблон, исключения, сезонность, локации и затрагиваемое оборудование. Справа отображается preview ближайших применений и конфликтов. Timezone всегда виден.

### 21.3. Ручное переопределение

Диалог требует объект, новое состояние, срок окончания и причину. Пользователь видит, какое расписание временно перекрывается и когда управление вернётся к автоматическому режиму.

### 21.4. Сценарий управления

Сценарий показывается линейным списком шагов с группами параллельного выполнения. Для каждого шага видны команда, условие успеха, timeout и действие при ошибке. Runtime показывает состояние каждого шага; общий зелёный итог не появляется при частичном результате.

## 22. Отчёты, документы и сменный журнал

### 22.1. Отчёты

Маршрут: `/reports`.

Экран содержит:

- каталог шаблонов;
- «Недавние»;
- «Мои расписания»;
- поиск и категории;
- предварительный просмотр параметров справа.

Основной сценарий: `контекст или сохранённый фильтр → preview → сформировать сейчас или запланировать`. Переход из оборудования, событий и ТОиР переносит объект, локацию и период.

Полный BI-конструктор не размещается на операторском экране. Расширенный конструктор, если он существует, открывается как отдельный editor workspace.

### 22.2. Документы

Документы доступны из контекста связанного объекта. Общий каталог поддерживает поиск, тип, срок действия и владельца. Действующая версия обозначена явно; история открывается по запросу.

### 22.3. Сменный журнал

Маршрут: `/shift-log`.

При приёме смены показывается одна сводка:

- активные критические ситуации;
- неподтверждённые аварии;
- оборудование без связи;
- обслуживание и подавления;
- ручные режимы;
- незавершённые задачи;
- комментарии предыдущей смены.

Оператор подтверждает приём. В рабочем журнале автоматические факты и человеческие комментарии имеют разные типы. Передача смены формирует список незакрытого; исключить критический пункт без причины нельзя.

## 23. ТОиР и ППР

### 23.1. Навигация сервиса

Маршрут: `/maintenance`.

Подразделы раскрываются внутри пункта ТОиР в левой навигации, не создавая второй постоянный header:

- Обзор;
- Объекты обслуживания;
- Планы ППР;
- Календарь;
- Заявки и дефекты;
- Наряды;
- Регламенты;
- Ресурсы;
- Отчёты.

### 23.2. Обзор

Обзор содержит не более нескольких действительно рабочих показателей:

- просрочено;
- сегодня;
- ближайшие критические работы;
- требует назначения;
- ожидает приёмки;
- нет материалов или допуска.

Каждый счётчик фильтрует одну очередь ниже, а не ведёт на несвязанный дашборд. Десятки KPI и декоративные диаграммы не используются.

### 23.3. Объекты обслуживания

Основной экран — working list. В строке и карточке явно показывается один из статусов:

- только объект ТОиР;
- связан с устройством диспетчеризации;
- связь требует проверки.

Отсутствие телеметрии не выглядит ошибкой. Полная карточка содержит паспорт, документы, планы, работы, историю и опциональную связь с устройством.

### 23.4. План ППР

Форма плана разделена на:

- объект и назначение;
- способ запуска;
- периодичность/наработку/условие;
- календарь и окно выполнения;
- регламент и чек-лист;
- срок предварительного формирования наряда и правило при незавершённом предыдущем наряде;
- исполнителей, материалы, инструменты и внешние услуги;
- требования безопасности, допуска и приёмки;
- уведомления;
- preview ближайших дат.

Редкие комбинированные условия находятся в «Дополнительно». План можно приостановить с причиной и датой пересмотра. Изменение действующего плана показывает влияние на уже созданные и будущие наряды.

### 23.5. Календарь ППР

Основные режимы — календарь и список. Календарь поддерживает месяц, неделю и день; квартал/год открываются как временная шкала. Дополнительное представление показывает загрузку исполнителей и ресурсов без создания отдельного планировщика.

Цвет не является единственным признаком. Различаются прогноз, созданный наряд, назначено, выполняется, выполнено, просрочено, перенесено и отменено. Клик открывает правую карточку; drag переноса разрешён только при наличии права и всегда запрашивает причину.

Слева по запросу открывается backlog, справа — выбранная работа. Одновременно обе панели не закрепляются по умолчанию.

### 23.6. Конфликты

Планировщик показывает отсутствие исполнителя, допуска, материала, доступа или окна остановки. Конфликт содержит прямой переход к исправлению и не выражается только красной точкой.

## 24. Заявки, дефекты и наряды

### 24.1. Единый паттерн списка

Каждая сущность имеет saved views и правую карточку. Списки не превращаются в редактируемую электронную таблицу. Inline разрешается только для статуса, приоритета, исполнителя и срока при наличии права.

### 24.2. Заявка

Короткая форма содержит источник, объект/локацию, описание, приоритет, желаемый срок и вложения. Перед сохранением показываются похожие открытые заявки.

Создание из события автоматически переносит контекст, значения, временной интервал и ссылку на событие. Пользователь не заполняет их повторно.

### 24.3. Наряд

Маршрут: `/maintenance/work-orders/{id}`.

Полная карточка содержит:

- компактный status path;
- объект и локацию;
- сроки и исполнителей;
- регламент и чек-лист;
- безопасность и допуски;
- материалы и трудозатраты;
- измерения и фотографии;
- дефекты и связанные работы;
- активность и аудит.

Основное следующее действие зависит от состояния и находится в локальной toolbar. Остальные действия — в меню.

Базовый жизненный цикл: `Черновик → Запланирован → На согласовании (если требуется) → Назначен → Принят → В работе ↔ Приостановлен → На приёмке → Выполнен → Закрыт`. `Отклонён` и `Отменён` требуют причины; допустимые переходы зависят от политики и роли. Primary action всегда соответствует следующему допустимому переходу: согласовать, принять, начать, приостановить/возобновить, передать на приёмку, принять результат или закрыть.

Для внешнего исполнителя используется ограниченное рабочее место без доступа к общему shell: открыть назначенный наряд по безопасной ссылке/QR, принять, вести чек-лист и время, приложить фото, зафиксировать дефект и передать результат. Подпись и обязательность приёмки включаются политикой конкретной работы.

### 24.4. Чек-лист

Чек-лист использует вертикальный поток. Шаг показывает тип ответа, обязательность и состояние. Условные шаги раскрываются только после соответствующего ответа. Невалидное значение объясняет допустимый диапазон и не удаляет ввод.

### 24.5. Мобильное рабочее место техника

На узком viewport показываются только назначенные работы, инструкции, чек-лист, фотографии, измерения, материалы и завершение. Полный desktop shell заменяется компактной навигацией. Состояние несинхронизированных данных постоянно видно; конфликт не разрешается молча.

## 25. «Моя работа», поиск и избранное

### 25.1. Моя работа

Маршрут: `/my-work`.

Основной переход выполняется из avatar menu; также допускаются прямые и контекстные ссылки. Отдельного пункта в global header или основной левой навигации нет.

Экран содержит четыре фильтрующих счётчика:

- просрочено;
- сегодня;
- требует решения;
- назначено мне.

Ниже расположена единая очередь задач из событий, инцидентов, согласований, нарядов, заявок/дефектов, работ на приёмке, публикаций и истекающих допусков. Тип задачи виден, но каждая служба не создаёт отдельный мини-дашборд.

Строка показывает тип, краткое действие, объект/локацию, приоритет, срок, состояние и назначившего. Доступны фильтры по типу, сроку, состоянию, локации и роли пользователя, а также группировка по сроку или типу. Клик открывает правую карточку с контекстом, историей и одним следующим действием. В зависимости от процесса доступны **«Принять»**, **«Передать»** и **«Не могу выполнить»**; передача и отказ требуют адресата/причины и не исчезают из аудита.

### 25.2. Глобальный поиск

Открывается из header или `Ctrl+K`. Поиск охватывает только разрешённые сущности и группирует результаты:

- локации;
- оборудование;
- параметры и команды по понятному имени, без запуска команды из поиска;
- дашборды;
- экраны и мнемосхемы;
- события и инциденты;
- объекты ТОиР, планы ППР, регламенты, заявки, дефекты и наряды;
- отчёты и документы;
- сотрудники с переходом на `/users/{userId}`;
- административные сущности при наличии права.

Результат показывает понятное имя, тип, локацию и состояние. Технический идентификатор доступен вторично. Результат по сотруднику показывает только разрешённую рабочую идентичность; скрытые контакты, отсутствие, график и организационные сведения не попадают ни в карточку результата, ни в подсказки. Поиск не раскрывает существование недоступных объектов.

### 25.3. Быстрые действия

Командная палитра позволяет открыть сервис, объект, представление или создать заявку. Технологическая команда через неё не запускается.

### 25.4. Избранное и недавнее

Избранными могут быть постоянные URL. Пользователь группирует их в папки; подборку можно опубликовать группе при наличии права, не раскрывая недоступные элементы. Недавнее хранит рабочий контекст, но при повторном открытии заново проверяет права. Избранное управляется в `/settings/favorites`, доступно из поиска и может выводиться компактным блоком на Рабочем столе; «Моя страница» не является центром управления избранным.

## 26. Kiosk, wallboard и терминалы

### 26.1. Kiosk

Kiosk скрывает общий shell и показывает назначенный дашборд. Навигация ограничена экраном и профилем. Hover не требуется. Любое управление проходит через временный режим и effective command policy: запрет, ограниченное управление без персональной идентификации, PIN либо полная повторная авторизация сотрудника.

В kiosk отсутствуют global header, navigation rail, Диспетчер событий, личный inbox, avatar menu и произвольный переход к другим сервисам. Переход из kiosk в обычную shell не предоставляется; сервисный выход и управление доверенной device session находятся за пределами статического прототипа.

### 26.2. Wallboard

Wallboard по умолчанию read-only. Он может переключать экраны по плейлисту, останавливаться на критическом состоянии и возвращаться к расписанию после нормализации. Управляющие элементы в этом профиле отсутствуют.

### 26.3. Офлайн

При потере связи:

- появляется заметный overlay с временем последнего обновления;
- значения маркируются как устаревшие;
- команды блокируются;
- плейлист может продолжить показ только с явной отметкой offline;
- накопленные локальные нажатия не отправляются после reconnect.

### 26.4. Реестр терминалов

Маршрут: `/admin/terminals`.

Таблица показывает имя, локацию, профиль, дашборд, online, последнее появление, текущий экран, версию конфигурации, разрешение и синхронизацию. Клик открывает карточку с действиями перепривязки, блокировки и отзыва.

Несколько панелей могут использовать общий профиль, но каждая имеет отдельную device identity. Отдельная пользовательская учётная запись на каждый планшет не требуется.

### 26.5. Регистрация и effective profile

Новый терминал открывает `/terminal/enroll`, получает короткий одноразовый код и показывает состояние ожидания. Администратор вводит код в `/admin/terminals/enroll`, проверяет сведения устройства, назначает локацию, групповой профиль и стартовый дашборд. После подтверждения создаётся отдельная device identity; код становится недействительным. Состояния: **не зарегистрирован → ожидает подтверждения → активен → заблокирован/отозван**.

Общий профиль безопасности и стартовый дашборд назначаются независимо: терминалы с одним профилем могут открывать разные опубликованные дашборды. Blocked или revoked identity не получает содержимое назначенного дашборда.

Карточка показывает effective profile и источник каждого значения: организация, dashboard, групповой профиль терминала или локальное исключение. Профиль терминала задаёт kiosk-команды, PIN/повторную авторизацию, правила воспроизведения, расписание, timeout управления и offline-поведение и может только ужесточать organization/dashboard defaults. Дашборд и состав плейлиста назначаются отдельно от security profile.

В статическом прототипе `?terminal={id}` является только локальным selector административного preview. Он не аутентифицирует терминал и не фиксирует production-маршрут. В рабочем продукте device identity поступает из доверенной зарегистрированной сессии.

## 27. «Моя страница», страницы сотрудников и настройки профиля

### 27.1. Разделение сущностей

| Сущность | Назначение | Маршрут |
|---|---|---|
| **Главная / Рабочий стол** | Личная оперативная сводка для начала и продолжения работы | `/home` |
| **Моя страница** | Рабочая идентичность текущего сотрудника и доступный коллегам профессиональный контекст | `/me` |
| **Страница сотрудника** | Та же сущность при просмотре другого человека | `/users/{userId}` |
| **Настройки профиля** | Полноразмерный личный сервис изменения разрешённых данных и пользовательских предпочтений | `/settings/*` |

«Моя страница» не является дашбордом, стартовой страницей или контейнером настроек. Её нельзя скопировать, создать из шаблона либо произвольно перестроить. Собственная и чужая страницы используют одну композицию; различаются доступные данные и действия.

«Моя страница» и личные настройки открываются из avatar menu или прямой ссылкой и не занимают пункты основной левой навигации. Страница коллеги доступна из глобального поиска, назначения, комментария, события, наряда и других ссылок на сотрудника.

### 27.2. Состав страницы сотрудника

Страница строится вокруг человека и его рабочего контекста:

- фотография или инициалы, ФИО, должность, подразделение и рабочая локация;
- руководитель и место в организационной структуре;
- рабочие контакты и предпочтительный разрешённый способ связи;
- рабочий статус, доступность и срок действия статуса;
- смена, отсутствие, отпуск и замещающий сотрудник, если зрителю разрешено это видеть;
- зоны ответственности, специализация, квалификации и допуски без раскрытия ролей безопасности;
- доступные зрителю текущие задачи, наряды, инциденты и согласования;
- закреплённые рабочие материалы: дашборды, инструкции, регламенты, отчёты и документы;
- краткая человеческая рабочая активность и общий доступный контекст.

Пустые необязательные блоки скрываются. Композиция остаётся предсказуемой и не использует Dashboard Editor. Страница не запускает технологические команды и не заменяет административную карточку пользователя.

### 27.3. Собственная и чужая страница

На `/me` дополнительно доступны:

- изменение рабочего статуса с обязательным сроком или явным бессрочным состоянием, если это разрешено политикой;
- переход в соответствующий раздел «Настроек» для редактирования разрешённых полей;
- переход в `/settings/privacy` к режиму **«Посмотреть как сотрудник»**, показывающему effective visibility без раскрытия административных правил.

На `/users/{userId}` доступны только применимые действия: открыть разрешённый рабочий контакт, упомянуть сотрудника, назначить задачу или перейти к общему рабочему объекту. Внутренний чат не вводится только ради профиля; действие связи показывается при наличии соответствующей интеграции.

Статусы не смешиваются:

| Статус | Источник | Что означает |
|---|---|---|
| Присутствие | Система | Пользователь online/offline в приложении |
| Оперативная доступность | Пользователь или политика | Доступен, занят, не беспокоить, на объекте; имеет срок действия |
| Рабочее состояние | Смена, график, отсутствие | На смене, вне смены, отсутствует, в отпуске |

Online не доказывает нахождение на смене, а статус «не беспокоить» не изменяет обязательную маршрутизацию оповещений.

### 27.4. Владение полями и видимость

| Категория | Кто изменяет | Effective visibility |
|---|---|---|
| ФИО, должность, подразделение, руководитель | Организационный каталог или администратор | Обязательная организационная идентичность |
| Фото, краткое описание, специализация, предпочтительный контакт | Сотрудник в разрешённых пределах | Политика организации плюс допустимый выбор сотрудника |
| Статус, смена, отсутствие и замещение | Пользователь, расписание или workflow | По служебной роли и рабочему контексту |
| Задачи, работы, инциденты, материалы и активность | Предметные сервисы | Только если зритель уже имеет доступ к соответствующему объекту |
| Роли безопасности, effective permissions, личные каналы, сеансы и security events | Административные и личные настройки | Никогда не выводятся как содержимое страницы сотрудника |

Для read-only поля виден источник данных. Пользователь может сузить видимость только там, где это разрешено, но не расширить её сверх политики организации. Скрытие блока не является авторизацией: каждый запрос повторно проверяет право зрителя.

### 27.5. Закреплённое и рабочая активность

«Закреплённое» содержит только ссылки на существующие опубликованные рабочие сущности. Закрепление не копирует объект, не меняет его владельца и не предоставляет доступ. Недоступный зрителю объект отсутствует без раскрытия названия, счётчика или причины скрытия.

Рабочая активность ограничена понятными совместными результатами: опубликована инструкция или дашборд, принято назначение, добавлен рабочий комментарий, завершена доступная работа. Raw audit, попытки входа, изменения прав, технические события и скрытые объекты здесь не показываются. Автоматический блок `Works with` не строится по закрытым материалам и переписке.

### 27.6. Полноразмерный сервис «Настройки профиля»

Базовый маршрут: `/settings`. Действие **«Настройки профиля»** находится в avatar menu; сервис открывается полноценной страницей в рабочей области и также доступен по прямой ссылке. На основном профиле применяется компактная локальная боковая навигация. Modal используется только для короткого подтверждения или preflight и не содержит весь процесс настройки.

Основные разделы:

- `/settings/profile` — разрешённые данные профиля и источники read-only полей;
- `/settings/privacy` — preview «как меня видят» и допустимая видимость полей;
- `/settings/home` — разрешённая персонализация Рабочего стола и сброс к компоновке организации;
- `/settings/interface` — язык, timezone, тема, плотность, единицы, контрастность и поведение панелей;
- `/settings/security` — активные сеансы, MFA и пароль, если ими не управляет внешний identity provider;
- `/settings/favorites` — избранные и недавние ссылки;
- `/settings/notifications/*` — effective policy, подписки, расписание отсутствия и каналы.

Настройки конкретных таблиц остаются в saved views. Системные параметры, пользователи, роли, политики и терминалы находятся в `/admin/settings` и связанных административных маршрутах и не дублируются в личных настройках.

### 27.7. Граница социальных механик

Dispatcher использует только полезные элементы социальной модели: узнаваемую страницу человека, рабочий статус, профессиональный контекст, закреплённые материалы, упоминания и переходы к разрешённой совместной работе.

Не вводятся стена общего назначения, друзья, подписчики, лайки, репосты, рейтинги, достижения, публичная индексация, произвольная пользовательская лента и копирование страниц. Комментарии и упоминания принадлежат предметным объектам, а не социальной стене пользователя.

## 28. Администрирование и здоровье платформы

### 28.1. Структура

Базовый маршрут: `/admin`; пункт **«Настройки»** в левой навигации открывает `/admin/settings`.

Основные разделы:

- локации и организационная структура;
- пользователи, группы и роли;
- подключения и источники;
- библиотеки интерфейса;
- политики событий, команд и оповещений;
- календари и смены;
- терминалы;
- здоровье платформы;
- качество данных;
- аудит.

Пункт и раздел системных настроек видны только администраторам и полностью скрыты для остальных пользователей; прямой URL повторно проверяет право.

### 28.2. Пользователи и права

Административная карточка учётной записи находится в `/admin/users` и не совпадает со страницей сотрудника `/users/{userId}`. Она доступна только уполномоченным администраторам. Основной путь назначения — группа/роль плюс область локаций. Карточка учётной записи показывает:

- итоговые доступные сервисы;
- области объектов;
- действия;
- источник каждого права;
- временные назначения;
- конфликты и истекающие права.

Отдельный режим отвечает на вопросы «Что доступно пользователю?» и «Кто имеет доступ к объекту?». Плоская матрица из тысяч checkboxes не является основным интерфейсом.

Создание пользователя, включение в группы, назначение роли/области, временный доступ и делегирование администратора локации оформляются короткими последовательными формами с effective-permissions preview. Изменение роли показывает затронутых пользователей и объекты до сохранения; удаление последнего администратора допустимой области блокируется.

Роли безопасности, effective permissions, конфликты прав и временные назначения не выводятся на «Моей странице» или странице коллеги. Профессиональная роль, должность и зона ответственности не используются как замена авторизации.

### 28.3. Наследуемые настройки

Для значения показываются источник, наследование и локальное переопределение. Перед глобальным изменением интерфейс показывает количество затрагиваемых объектов и исключения. Массовое применение имеет preview.

### 28.4. Здоровье платформы

Маршрут: `/admin/health`.

Экран использует рабочий список и сводные счётчики:

- источники и подключения;
- устройства без связи;
- ошибки конфигурации;
- доставка оповещений;
- терминалы;
- фоновые операции;
- публикации;
- системные ограничения ресурсов.

Выбор проблемы открывает причину, затронутую область, последнее успешное состояние и переход к диагностике.

### 28.5. Центр качества данных

Маршрут: `/admin/data-quality`.

Категории:

- устаревшие и плохие данные;
- пропуски истории;
- единицы и диапазоны;
- несопоставленные точки;
- расхождения с шаблоном;
- дубли адресов;
- нестабильные подключения;
- рассинхронизация времени.

Счётчик открывает тот же список с фильтром. Проблеме можно назначить ответственного и отслеживать устранение.

### 28.6. Активность и аудит

Рабочие комментарии, человеческая активность и неизменяемый аудит разделяются. На странице сотрудника может отображаться только понятная рабочая активность по объектам, уже доступным зрителю. Технические изменения, security events, входы, изменения прав и raw audit там не показываются. В административной карточке технические изменения включаются фильтром; полный аудит доступен отдельному праву и поддерживает поиск по пользователю, объекту, времени, действию и результату.

### 28.7. Общий цикл публикации

Для дашбордов, мнемосхем, аварийных правил, расписаний, политик оповещений и регламентов используется один интерфейсный цикл:

`Черновик → Проверка → Preview → Анализ влияния → Согласование при необходимости → Публикация → Наблюдение → Откат новой версией`.

Состояние черновика и опубликованная версия всегда различимы. Публикация является целостным действием; открытый операторский экран не получает половину старой и половину новой конфигурации.

Простые snapshot-шаблоны устройств создаются, выбираются и удаляются в универсальном редакторе устройств согласно § 16.8. Они не образуют отдельный административный сервис; централизованный governance и версионирование могут быть добавлены позднее отдельным решением.

## 29. Вход, ошибки доступа и восстановление сеанса

### 29.1. Вход

Страница входа содержит знак `D`, название Dispatcher, основной способ входа и ссылку на помощь. Если настроен единый вход, он является основным действием; локальная форма раскрывается только при необходимости.

Форма не показывает технические сведения сервера. Ошибка не сообщает, существует ли введённая учётная запись. Клавиатурный фокус, autocomplete и менеджеры паролей поддерживаются корректно.

### 29.2. Дополнительное подтверждение

MFA или повторная авторизация выполняются отдельным коротким шагом. Интерфейс объясняет, что подтверждается: вход, включение управления или конкретная критическая команда.

### 29.3. Истечение сеанса

Перед истечением активного сеанса пользователь получает предупреждение. Если срок истёк:

- просмотр блокируется безопасным overlay;
- черновик сохраняется локально настолько, насколько это разрешено;
- после входа пользователь возвращается к предыдущему URL;
- технологическая команда не повторяется автоматически;
- выполнявшаяся команда после входа показывается по её фактическому состоянию.

### 29.4. Нет доступа и объект не найден

Состояния «нет права», «объект удалён/архивирован» и «ссылка неверна» различаются. Пользователю не раскрываются данные недоступного объекта, но предлагается безопасный возврат к родительскому сервису.

## 30. Главная / Рабочий стол и профильные рабочие места

### 30.1. Главная / Рабочий стол

Маршрут: `/home`.

Главная является независимым оперативным Рабочим столом пользователя. Она всегда открывается по `/home`, не является «Моей страницей», псевдонимом последнего рабочего дашборда и не перенаправляет на `/d/{dashboard}/{screen}`.

Начальную компоновку назначает организация по роли и контексту эксплуатации. Пользователь не выбирает именованный шаблон Рабочего стола. В `/settings/home` он может в разрешённых пределах добавлять, скрывать и переставлять доступные виджеты, управлять избранными переходами и восстановить исходную компоновку организации.

Копирование страницы другого сотрудника и создание Рабочего стола из неё отсутствуют.

Рекомендуемая исходная компоновка Рабочего стола содержит:

- 3–5 показателей, реально определяющих начало работы;
- одну приоритетную очередь;
- избранные и недавние переходы;
- при необходимости краткую информацию о смене в содержательном виджете;
- не более нескольких быстрых действий.

Содержание зависит от роли:

| Роль | Основной акцент |
|---|---|
| Диспетчер | Передача смены, события, недоступные объекты, активные инциденты |
| Инженер | Подключения, качество данных, черновики и публикации |
| Техник | Сегодня, просрочено, назначенные наряды |
| Планировщик | Календарь ППР, backlog, ресурсы и конфликты |
| Руководитель | Отклонения, риски, время реакции и выполнение ТОиР |

Отдельный приветственный блок с именем пользователя не используется: активный пользователь отображается в global header.

### 30.2. Специализированные рабочие места

Климат, энергетика, пожарная безопасность, охрана, ИТ и руководительские обзоры реализуются как типовые дашборды, сохранённые представления и быстрые переходы. Они не получают отдельную несовместимую оболочку и не дублируют реестры.

## 31. Доступность, клавиатура и touch

### 31.1. Базовые требования

- Все основные функции доступны с клавиатуры.
- Фокус видим во всех темах.
- Порядок фокуса следует визуальному порядку.
- Иконка без подписи имеет доступное имя и tooltip.
- Цвет не является единственным признаком.
- Масштабирование браузера не скрывает критические действия.
- Таблицы имеют корректные заголовки и читаемое состояние выбранных строк.
- Обновление realtime не перехватывает фокус и не объявляет каждое изменение без необходимости.

### 31.2. Клавиатурные действия

Минимальный общий набор:

| Действие | Клавиша |
|---|---|
| Глобальный поиск | `Ctrl+K` |
| Локальный поиск списка | `/`, если фокус не в поле ввода |
| Закрыть panel/dialog | `Esc` |
| Справка по клавишам | `?` |
| Перемещение по строкам | `↑` / `↓` |
| Открыть выбранное | `Enter` |
| Выбрать строку для bulk | `Space` |
| Перейти к следующему основному региону | стандартная навигация `Tab`/landmarks |

Команды оборудования не получают одноклавишный глобальный shortcut.

### 31.3. Touch — перспективное развитие

Требования этого подраздела сохраняются для будущего touch-этапа и не входят в текущую приёмку профиля `1920×1080`.

- Значимое действие не зависит от hover, right-click, double-tap или long press.
- Touch-зоны соответствуют профилю и имеют расстояние между опасными действиями.
- Pinch применяется для карты, мнемосхемы и тренда.
- Вертикальный жест страницы не перехватывается маленькой мнемосхемой без явного фокуса.
- Критическая команда всегда имеет видимую кнопку и подтверждение по политике.

### 31.4. Звук

Звуковое оповещение всегда сопровождается визуальным индикатором. Пользователь видит источник звука и правило его отключения. Личное приглушение не должно нарушать обязательную критическую политику рабочего места.

Маршруты являются продуктовой картой, а не спецификацией backend routing.

| Сервис | Основные маршруты |

`/home`, `/me` и личные `/settings/*` имеют разные назначения и не перенаправляют друг на друга. Возврат к последнему активному дашборду является поведением пункта основной навигации и не меняет канонические URL. `/admin/users` остаётся административной областью и не заменяет `/users/{userId}`.

## 35. Состав текущего HTML/CSS/JS-прототипа

Текущий статический прототип проверяет общую продуктовую логику на локальных данных. Он не является промышленным frontend и не определяет backend-контракты самостоятельно.

### 35.1. Представленный Core-контур

1. Глобальная оболочка и оперативный Рабочий стол `/home` с назначенной организацией компоновкой и ограниченной персонализацией.
2. Каталог дашбордов, первое открытие каталога, возврат к последнему активному дашборду и явное действие «Все дашборды».
3. Runtime дашборда с адресуемыми окнами `Combined`, `Mimic`, `Widgets` и рабочим fullscreen-mimic состоянием.
4. Диспетчер событий с фильтрами, realtime/history и правой карточкой.
5. Реестр оборудования и краткая карточка.
6. «Мои оповещения» с четырьмя устойчивыми подразделами.
7. «Моя работа» с принятием/передачей задачи.
8. Два независимых demo-маршрута Dashboard Editor и SVG Editor внутри общей app shell с активным пунктом «Конструктор» и компактной editor toolbar, без полноценного графического редактирования.
9. «Моя страница» `/me`, representative-страница коллеги `/users/{userId}` и полноразмерные разделы `/settings/profile`, `/settings/privacy` и `/settings/home`.
10. Avatar menu, личный inbox веб-оповещений и разделение личных `/settings/*` от admin-only `/admin/settings`.

Для `Combined` в прототипе мнемосхема по умолчанию использует `clamp(320px, 55vh, 640px)`, затем следует widget grid и одна прокрутка страницы. Это эталон для проверки, а не обязательный размер опубликованных дашбордов.

Текущий прототип проектируется и проверяется для `1920×1080`. Адаптивные viewport, touch и реализация wheel-zoom/pan ЛКМ не входят в этот контур; последние остаются итоговым требованием runtime из § 12.5.

### 35.2. Представленный расширенный логический контур

- универсальный редактор устройств с ручным созданием строк и загрузкой CSV в общий staging; в тестовом прототипе только Modbus TCP и SNMP;
- отдельная история тега с переходом к исходному дашборду;
- обзор ТОиР, календарь ППР и карточка наряда;
- representative-экраны администрирования пользователей/прав, подключений, здоровья, качества данных и аудита;
- независимые lifecycle Dashboard Editor и SVG Editor с проверкой, публикацией и откатом;
- регистрация терминала, отдельная device identity, effective profile и kiosk-вариант назначенного дашборда;
- административная политика оповещений с effective-policy preview;
- упрощённая связь события с отдельным инцидентом и заявкой без изменения alarm occurrence.

Этот контур демонстрирует только состояния и переходы. Реальные IAM, persistence, realtime, публикация, pairing и физические команды остаются backend-задачами.

### 35.3. Обязательные интерактивные демонстрации Core

Комбинированный демонстрационный дашборд ограничивается пятью предметными элементами: состояние оборудования, короткий тренд, компактный список событий, навигация между окнами и command preview; мнемосхема выступает основной визуальной областью, а не шестым информационным виджетом.

- свернуть/раскрыть навигацию, открыть фильтры и подробности без потери состояния;
- сменить локацию, открыть глобальный поиск и прямую ссылку;
- включить временное управление, увидеть таймер и открыть command preview без отправки;
- применить saved view; обновить видимую строку realtime без скачка; показать «Новых: N»;
- отдельно выполнить acknowledgement и assignment; потребовать комментарий по policy;
- shelve/unshelve, показать expiry, suppression-причину и bulk preflight с частичным результатом;
- открыть read-only History и перейти к актуальному occurrence;
- показать effective notification policy, запрет отключения обязательного канала и отпуск с принятием замещения;
- масштабировать мнемосхему командами `100%`, «Вписать», «По ширине», «Сбросить к авторскому виду»;
- прокрутить combined/widget dashboard одной вертикальной прокруткой;
- выбрать оборудование на мнемосхеме и синхронно обновить faceplate, короткий тренд и связанные события без автоматического выполнения команды;
- переключить `Live`/`History`, постоянно показать активный режим и заблокировать command preview в историческом режиме;
- показать `Good`, `Uncertain`, `Bad`, `Stale`, `Offline` и `Initializing` единообразно в виджете, символе и faceplate;
- открыть «Мою работу» из avatar menu, затем принять, передать или выбрать «Не могу выполнить» с обязательной причиной;
- открыть оба редактора с сохранёнными global header и navigation rail, включить «Только холст» и отдельно «Полноэкранный холст»;
- при уходе из редактора сохранить черновик либо получить предупреждение о несохранённых изменениях;
- для обоих редакторов проверить только точную сохранённую revision, после изменения снова заблокировать публикацию до сохранения и повторной проверки, а откат выполнить как новую revision;
- открыть каталог без сохранённого дашборда, выбрать дашборд, перейти в другой сервис, вернуться к последнему активному и затем явно открыть «Все дашборды»;
- открыть через avatar menu «Мою страницу», «Мою работу», настройку оповещений и полноразмерные настройки профиля;
- открыть личный inbox, увидеть unread и перейти к доступной связанной сущности;
- перейти с «Моей страницы» в `/settings/profile` для изменения разрешённого поля, увидеть источник read-only поля, затем открыть preview в `/settings/privacy`;
- открыть страницу коллеги с viewer-filtered данными и без раскрытия недоступных работ;
- изменить разрешённый состав Рабочего стола в `/settings/home` и восстановить компоновку организации.

### 35.4. Проверки расширенного логического контура

- провести наряд через допустимые состояния, заполнить чек-лист и передать на приёмку;
- открыть расширенные настройки безопасности и завершить другой сеанс;
- зарегистрировать терминал по одноразовому коду, подтвердить назначение, показать effective profile, затем заблокировать или отозвать device identity;
- выполнить preview обязательной политики: показать итоговых получателей, каналы, времена, contributing rules и конфликт, блокирующий публикацию;
- добавить ручную строку и загрузить CSV в общий staging, отредактировать строку в правой форме, показать `create/update/skip`, блокирующую ошибку структуры и неблокирующие результаты проверки связи и тестового опроса;
- открыть работу из календаря ППР и её правую карточку.

### 35.5. Что прототип не обязан реализовывать

- backend и постоянное хранение;
- настоящее подключение оборудования;
- реальную аутентификацию;
- фактическую отправку команд;
- полноценный drag-and-drop редактор;
- импорт или редактирование SVG-path;
- реальные каналы оповещений;
- генерацию отчётов;
- offline-синхронизацию.

Демонстрационные данные должны включать норму, аварию, плохое качество, stale, offline, обслуживание и отсутствие права.

## 36. Критерии проверки интерфейса

### 36.1. Рабочая область

- На desktop присутствует ровно один постоянный global header.
- Постоянный footer отсутствует.
- Дашборд, мнемосхема и тренд могут использовать всю оставшуюся высоту.
- Левая и правая панели закрываются.
- Панели переходят в overlay до сжатия центральной области ниже минимума.
- На профиле `1920×1080` нет горизонтального scroll viewport.
- Локальная toolbar не дублирует глобальный header.
- Header и editor/event toolbar не переносятся на вторую строку.

### 36.2. Простота

- На экране есть одно очевидное основное действие.
- Редкие параметры скрыты в «Дополнительно» или контекстной панели.
- Один и тот же список использует одинаковые фильтры во всех сервисах.
- Runtime не показывает инструменты редактирования.
- Обычному пользователю не показываются технические выражения и bindings.
- На главной нет набора из десятков несвязанных виджетов.

### 36.3. Дашборд и мнемосхема

- Текущая проверка выполняется на основном профиле `1920×1080`; остальные viewport относятся к перспективному этапу.
- Виджеты переносятся и продолжаются вниз.
- В `Widgets` и `Combined` основной scroll принадлежит странице; в `Mimic` page scroll отключён в пользу zoom/pan.
- Итоговый runtime поддерживает wheel-zoom относительно курсора и pan перетаскиванием ЛКМ как на карте; текущий HTML-прототип может ограничиваться кнопками масштаба и визуальной демонстрацией.
- Маленькая мнемосхема не блокирует обычную прокрутку страницы.
- Zoom controls закреплены относительно embedded-контейнера, а в рабочем полноэкранном режиме — относительно viewport.
- Встроенная мнемосхема до явного focus не перехватывает wheel у страницы; полноэкранный `Mimic` масштабируется wheel непосредственно.
- `100%` является нормализованным масштабом `1.0`, а «Сбросить к авторскому виду» восстанавливает опубликованные scale и position.
- Контекст окна занимает одну компактную строку либо закрываемую панель и не образует ещё один header.
- `Live` и `History` различимы постоянно; в `History` команды недоступны, а локальный период виджета обозначен как override.
- Явный выбор объекта синхронизирует faceplate и связанные виджеты; первый объект не выбирается скрыто и widget-to-widget цепочка не меняет контекст молча.
- Первый клик по символу выбирает объект, но не выполняет команду и не переназначает target уже открытого command dialog.
- `Good`, `Uncertain`, `Bad`, `Stale`, `Offline` и `Initializing` имеют одинаковую семантику во всех dashboard-компонентах.
- Fullscreen-overlay сохраняет выход, связь, критический счётчик и индикатор управления, даже когда остальная оболочка скрыта.

### 36.4. События

- В одном сервисе доступны события и аварии.
- Severity, active/cleared, acknowledgement и assignment отображаются независимо.
- Для события и alarm occurrence доступны только применимые колонки и действия.
- Новая запись не сбрасывает выбор и прокрутку.
- Изменение уже видимой записи обновляется на месте; вставка новых строк замораживается во время работы оператора.
- Карточка открывается справа без потери списка.
- Подтверждение не закрывает и не назначает событие автоматически.
- History read-only и позволяет перейти к актуальному occurrence.
- Shelving требует причины и срока; unshelve и expiry возвращают активную аварию в очередь.
- Suppression имеет видимое объяснение и не удаляет запись из истории.
- Bulk action показывает scope, eligible/ineligible записи и частичный результат.

### 36.5. Оповещения

- Обязательные и дополнительные правила различаются.
- Пользователь видит effective policy.
- Отпуск с обязательной обязанностью не активируется до принятого покрытия или административного override.
- Канал можно подтвердить и протестировать.
- Обязательный канал нельзя отключить без рабочего допустимого замещения.
- Критическое обязательное уведомление нельзя выключить личным quiet hours.
- Административная эскалация отображается линейными шагами и явно указывает условие остановки; delivery receipt не считается acknowledgement.
- Перед публикацией preview показывает итоговых получателей, каналы, времена, причины выбора и конфликты нескольких обязательных правил.

### 36.6. Управление

- По умолчанию управление выключено.
- Активный режим постоянно виден и имеет timeout.
- Команда повторно проверяет состояние и качество.
- Успех не показывается до предусмотренной обратной связи.
- Неопределённый результат отличается от успеха и ошибки.
- Текущий фиксированный Kiosk применяет профильную политику PIN либо ограниченного управления без персональной идентификации; это не изменяет правила обычного профиля `Operator`.

### 36.7. Достоверность и доступность

- Stale и bad value не выглядят актуальными.
- Цвет не является единственным признаком.
- Клавиатурный фокус виден.
- На перспективном touch-этапе действия не зависят от hover/right-click.
- Realtime не перехватывает фокус.
- Нет права, нет данных и ошибка различаются.

### 36.8. Рабочие сценарии Core prototype

- «Моя работа» показывает предметный контекст и поддерживает принять, передать и отказ с причиной.
- «Моя работа» отсутствует в global header и основной левой навигации и доступна из avatar menu, прямых и контекстных ссылок.
- Dashboard Editor и SVG Editor имеют разные маршруты и редактируют разные сущности внутри общей shell; «Конструктор» активен.
- Global header и navigation rail скрываются в редакторе только явной командой «Полноэкранный холст».
- Уход из редактора сохраняет черновик либо предупреждает о ещё не сохранённых изменениях.
- Dashboard Editor и SVG Editor публикуют только точную сохранённую и проверенную revision; любое изменение после проверки снова блокирует публикацию, а откат создаёт новую revision.
- Прямая ссылка и Back/Forward восстанавливают список, выбранную запись и панель.

### 36.9. «Моя страница» и Настройки

- `/home`, `/me`, `/users/{userId}` и `/settings/*` имеют разные назначения и не подменяют друг друга.
- Avatar menu содержит «Моя страница», «Моя работа», «Настройка оповещений», «Настройки профиля» и «Выйти».
- Личные `/settings/*` занимают полноценную рабочую область; `/admin/settings` показывается в левой навигации только администраторам.
- Личный inbox имеет отдельную иконку и unread, не подменяет Диспетчер событий, а прочтение не выполняет acknowledgement.
- Собственная и чужая страницы используют одну композицию; собственная добавляет только разрешённое изменение статуса и переходы к редактированию или preview в «Настройках».
- Для read-only поля виден источник, а пользователь не может расширить видимость сверх политики организации.
- Работы, материалы, закрепления и активность фильтруются по правам зрителя и не предоставляют новых прав.
- Страница сотрудника не копируется, не создаётся из шаблона и не редактируется Dashboard Editor.
- Присутствие, оперативная доступность и рабочее состояние отображаются как разные статусы.
- Технический аудит, security events и effective permissions не выдаются за рабочую активность сотрудника.

### 36.10. Редактор устройств

- Ручное добавление и загрузка файла CSV создают строки в одном staging без отдельных одиночного и пакетного режимов; выбор строки открывает справа полную форму с локальными ошибками полей.
- Тестовый прототип предлагает только Modbus TCP и SNMP с протокол-специфичными секциями; неподдерживаемый протокол остаётся ошибкой строки и не преобразуется в поддерживаемый.
- Новые строки получают `create`, совпадения — `skip/review`, `update` требует явного выбора администратора; `delete` отсутствует, а отсутствие строки в CSV ничего не удаляет.
- Только ошибки структуры и уникальности блокируют применение соответствующей строки. Проверка связи и тестовый опрос информативны, не обязательны и не создают статус «Не проверено».
- Копирование поддерживает количество экземпляров; автоматически инкрементируется только Modbus Unit ID, но не IP-адрес.
- Шаблон исключает ID, имя, IP-адрес, Modbus Unit ID и секреты; он применяется к выбранной staging-строке либо создаёт новый черновик, если строк нет, и не изменяет созданные устройства.
- SNMP v2c community по умолчанию равен `public`, маскируется и имеет показ; предупреждение не блокирует. SNMP v3 настраивается отдельно.
- Секреты отсутствуют в preview, результатах тестов и журналах интерфейса; отдельная ссылка на хранилище секретов не обязательна для тестового прототипа.

### 36.11. Kiosk и терминалы

- `/admin/terminals` использует обычную административную shell, а enrollment и kiosk runtime не показывают global header, navigation rail или переходы к другим сервисам.
- Общий профиль и стартовый дашборд назначаются независимо; каждая физическая панель имеет отдельную device identity.
- Одноразовый код после подтверждения не используется повторно; blocked или revoked identity не получает назначенный дашборд.
- PIN появляется только по политике профиля. При no-PIN сценарии аудит связывает действие с терминалом без ложного имени сотрудника.
- Offline немедленно блокирует управление; нажатия не ставятся в очередь и не воспроизводятся после reconnect.
- `?terminal={id}` считается только selector локального preview и не принимается за production-аутентификацию устройства.

## 40. Итоговая модель интерфейса

Dispatcher использует одну предсказуемую оболочку и несколько повторяемых типов рабочих экранов. В обычной эксплуатации пользователь видит данные и действия, а не инструменты настройки. Сложность переносится в контекстные панели, управляемые редакторы и административные preview, не уменьшая постоянную рабочую область.

Личный контур разделён явно: `/home` — оперативный Рабочий стол, `/me` — рабочая идентичность сотрудника, `/users/{userId}` — разрешённое представление коллеги, `/settings/*` — полноразмерные личные настройки, `/notifications` — личный inbox. Системные настройки находятся в `/admin/settings` и доступны только администраторам.

Целевая формула интерфейса:

**один компактный header + сворачиваемая навигация + максимальная рабочая область + одна контекстная панель + редакторы внутри общей shell с отдельным полноэкранным холстом по запросу.**

Текущие интерфейсные итерации ориентированы на `1920×1080`; общая адаптивность сохраняется как перспектива, а Kiosk представлен отдельным фиксированным логическим профилем. «Моя работа» относится к личному avatar menu. Dashboard runtime использует явный контекст, раздельные `Live/History`, единый selected object и целую published revision; Dashboard Editor и SVG Editor по умолчанию сохраняют global header и navigation rail, а полноэкранный холст включается отдельно. Универсальный редактор устройств объединяет ручное создание и загрузку CSV в одном staging; тестовый прототип ограничен Modbus TCP и SNMP. Итоговая мнемосхема использует wheel-zoom относительно курсора и pan ЛКМ. Диспетчер событий объединяет события и аварии, а личный inbox остаётся отдельным каналом адресованных пользователю веб-оповещений.
