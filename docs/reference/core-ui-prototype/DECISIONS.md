# Зафиксированные решения

- Название продукта: «Диспетчер».
- Логотип: буква `D`.
- Прототип работает на локальных тестовых данных.
- Один header высотой 48 px, navigation rail 56 px и overlay-карточка справа.
- Основная проверка выполняется в тёмной операторской теме.
- Текущий целевой viewport прототипа — `1920×1080`; адаптивные варианты на этом этапе не развиваются.
- Маршрутизация использует hash URL, чтобы `index.html` работал без сервера.
- Сервис называется «Диспетчер событий» и объединяет события и аварийные состояния.
- Управление включается отдельной кнопкой на ограниченное время.
- Повторная авторизация или PIN определяются политикой безопасности.
- Команды не отправляются: показывается только preflight/preview.
- Дашборд является отдельной сущностью с собственным контекстом и ссылкой.
- Первый вход в «Дашборды» открывает каталог; после выбора пункт навигации возвращает к последнему активному дашборду, а runtime содержит явную кнопку «Все дашборды».
- `/home` является независимым оперативным Рабочим столом с назначенной организацией компоновкой и ограниченной персонализацией через `/settings/home`.
- `/me` — «Моя страница» текущего сотрудника; `/users/{userId}` использует ту же композицию для страницы коллеги.
- `/settings/*` — отдельный полноразмерный сервис. Настройки не размещаются в modal, header или на странице сотрудника.
- Avatar открывает компактное меню личных маршрутов; `/settings/*` остаётся полноразмерной личной областью, а `/admin/settings` — отдельными системными настройками, видимыми только admin fixture.
- Все `/admin/*` повторно проверяют административное право; скрытый пункт навигации не считается авторизацией.
- Административная учётная запись отделена от страницы сотрудника: должность и профессиональная роль не предоставляют системных прав.
- Effective permissions показывают роль, область и источник назначения без плоской матрицы checkbox; изменение выполняется после impact preview, а последний администратор области защищён от снятия роли.
- Состояние платформы, качество данных, аварийный occurrence и технический аудит являются разными сущностями; аудит доступен только для чтения.
- Диагностика подключений в прототипе информационная, endpoint маскируется, секреты не отображаются и не попадают в аудит.
- Персональные оповещения показываются отдельной иконкой с unread badge и не смешиваются с общим Диспетчером событий.
- Компактный inbox ведёт в полный `/notifications`; read/unread относится только к доставленному сообщению, а переход к источнику повторно проверяет наличие доступной fixture-сущности.
- Wheel-zoom относительно курсора и перетаскивание мнемосхемы ЛКМ зафиксированы для итогового продукта, но намеренно не реализуются в текущем логическом прототипе.
- Страница сотрудника не копируется, не выбирается из шаблона и не редактируется Dashboard Editor.
- В global header на широком экране отображается имя «Алексей Смирнов»; отдельный приветственный header на главной не используется.
- Dashboard Editor и SVG Editor используют общий shell, но разные маршруты и сущности.
- Dashboard и SVG имеют независимые черновики и истории; сохранение не меняет runtime, а публикация применяет одну целую revision.
- Проверка относится к конкретному сохранённому черновику; несохранённые или изменённые после проверки данные публиковать нельзя.
- Откат не перезаписывает историю: выбранная старая revision становится основой новой опубликованной версии.
- Уход из редактора с локальными изменениями требует сохранить черновик, явно отказаться от изменений либо остаться.
- «Моя работа» не занимает global header и rail: маршрут доступен через avatar-menu и предметные ссылки, число задач показано badge.
- Счётчики «Моей работы» вычисляются из текущей очереди; переданная или возвращённая задача перестаёт считаться назначенной текущему пользователю, но сохраняется в общей истории прототипа.
- Личные подписки только расширяют обязательное покрытие; mandatory-каналы показываются read-only, а отпуск становится активным только после демонстрационного принятия замещения.
- `/maintenance` является отдельным эксплуатационным сервисом, а дашборд «ТОиР · Сегодня» его не заменяет.
- Объект обслуживания не обязан иметь телеметрию: состояния «только ТОиР», «связан с устройством» и «связь требует проверки» показываются явно.
- План ППР, прогноз календаря, наряд и личная задача имеют независимые состояния; переход наряда или создание заявки не выполняют acknowledgement события.
- Календарь текущей итерации ограничен неделей и списком без drag-and-drop; карточка наряда показывает только следующее допустимое основное действие.
- Editor routes сохраняют global header и rail; добавление оборудования выполняется одним stateful staging-экраном и меняет только локальные fixtures.
- Импорт оборудования по умолчанию только создаёт: existing требует явного admin-разрешения на update, skip игнорируется, отсутствующие строки ничего не удаляют.
- Только структурная валидность разрешает Apply; connection/poll информационные, независимые и никогда не являются gate.
- CSV выбирается реальным file input, читается на клиенте по header с comma/semicolon; неизвестный протокол не преобразуется в Modbus.
- Copy всегда спрашивает количество, не изменяет host и только для Modbus опционально создаёт последовательные Unit ID.
- Шаблоны создают редактируемый draft snapshot и исключают ID, имя, host, Unit ID, community и другие секреты; SNMP v2c по умолчанию использует masked `public` с предупреждением.
- Основной runtime имеет три окна с устойчивыми URL: `/d/main/overview`, `/d/main/mimic`, `/d/main/widgets`; список окон открывается в существующей правой панели.
- Выбор объекта хранится отдельно от панели и сохраняется между окнами; клик по символу только выбирает target и открывает faceplate.
- `History` всегда обозначен периодом и блокирует команды; временное управление ограничено текущим дашбордом, а preview фиксирует выбранный объект.
- В Диспетчере событий severity, condition, acknowledgement, assignment, shelving/suppression и связь с инцидентом остаются независимыми признаками.
- «История» событий полностью read-only; создание инцидента показано только как отдельная связанная сущность и пока не разворачивается в полноценную incident workspace.
- Администрирование терминалов остаётся внутри обычной shell на `/admin/terminals`; `/terminal/enroll` и kiosk runtime загружаются без global header и navigation rail.
- В прототипе `?terminal={id}` является только selector для административного preview у назначенного `/d/{dashboard}/{screen}`. Он не аутентифицирует device identity и не является зафиксированным production-маршрутом.
- Несколько панелей используют общий конфигурационный kiosk-профиль, но каждая имеет отдельную device identity для presence, блокировки и отзыва; отдельный пользователь на каждый планшет не создаётся.
- Общий профиль безопасности и стартовый дашборд назначаются независимо: несколько панелей с одним профилем могут открывать разные опубликованные дашборды.
- Политика профиля выбирает ограниченное управление без персональной идентификации либо PIN сотрудника. Только второй вариант связывает действие с человеком; реальная проверка PIN не реализована.
- Offline, блокировка, отзыв или выход из назначенного маршрута немедленно отключают временное управление. Нажатия offline не ставятся в очередь и не воспроизводятся после reconnect.
- Переход из kiosk в обычную shell отсутствует. Сервисный выход и доверенная device session остаются за пределами статического прототипа.
