# Диспетчер — Core UI prototype

Статический прототип по `../../WEB_INTERFACE_SPECIFICATION.md`. Сервер, сборка, база данных и подключение оборудования не требуются.

## Запуск

Откройте `index.html` напрямую в браузере или запустите каталог через Live Server в VS Code. Все подключения внутри прототипа относительные.

Текущая контрольная конфигурация — `1920×1080`. Адаптивная верстка, wheel-zoom и перетаскивание мнемосхемы будут проверяться на последующих этапах и в эту итерацию не входят.

## Core-контур

- компактная оболочка и оперативный Рабочий стол с ограниченной персонализацией;
- «Моя страница» и representative-страницы сотрудников;
- полноразмерные «Настройки» с профилем, видимостью, Рабочим столом, интерфейсом, безопасностью, избранным и оповещениями;
- avatar-menu с личными маршрутами, отдельный dropdown персональных оповещений и полный список `/notifications`; прочтение не подтверждает событие, а доступные ссылки ведут к событию, устройству, дашборду или истории тега;
- admin-only область `/admin/*`: общие системные настройки, учётные записи и роли, подключения, терминалы, состояние платформы, качество данных и read-only аудит;
- каталог дашбордов, избранное и возврат к последнему активному дашборду;
- основной дашборд с адресуемыми окнами `Combined`, `Mimic`, `Widgets` и рабочим полноэкранным режимом мнемосхемы;
- единый выбранный объект синхронизирует мнемосхему, faceplate, короткий тренд и связанные события;
- режимы `Live`/`History`: период виден постоянно, а command preview в истории заблокирован;
- единый Диспетчер событий: события и аварийные occurrence, рабочие фильтры, Realtime/История, acknowledgement, assignment, shelving и suppression;
- упрощённое создание отдельного инцидента из события с сохранением связи и без изменения состояния alarm occurrence;
- реестр оборудования и контекстная карточка;
- effective policy, добавление личной подписки, presets графика, отпуск с принятым замещением и переключаемые дополнительные каналы;
- единая очередь «Моя работа» с вычисляемыми счётчиками, фильтром типа, переходами к источникам и состояниями принятия, передачи и возврата;
- создание инцидента или заявки из события демонстрационно добавляет связанную задачу в «Мою работу», не изменяя acknowledgement события;
- сервис `/maintenance` с компактными разделами: обзор, независимые объекты обслуживания, планы ППР, недельный календарь, заявки/дефекты и наряды;
- объект ТОиР может существовать без устройства диспетчеризации; прогноз календаря, наряд и личная задача остаются разными сущностями;
- карточка наряда показывает один следующий переход, обязательный чек-лист и связи с планом, оборудованием и «Моей работой»;
- раздельные demo-маршруты Dashboard Editor и SVG Editor с независимыми черновиками, проверкой, impact preview, версиями, публикацией и откатом новой revision;
- `/my-work` доступен из avatar-menu и прямых ссылок; оба редактора работают внутри общей оболочки;
- `/equipment/add` — единый staging: ручные черновики и реальный выбор CSV без переключения single/batch;
- CSV читается в браузере по header с разделителем comma/semicolon; новые строки получают `create`, existing остаётся `skip/review` до явного admin-update, отсутствующие строки ничего не удаляют;
- структурная валидация — единственный gate для Apply; connection/poll остаются независимой информационной симуляцией;
- поддерживаются Modbus TCP и SNMP v2c/v3, копирование черновиков и безопасные шаблоны без ID, имени, host, Unit ID и секретов; v2c показывает masked `public` и предупреждение.
- временный режим управления в области текущего дашборда и command preview с зафиксированным target без отправки.
- `/admin/terminals` — реестр панелей с общими профилями, отдельными device identity, effective policy, перепривязкой, блокировкой и отзывом;
- `/terminal/enroll` и `/admin/terminals/enroll` показывают символическую регистрацию по одноразовому коду без backend;
- kiosk runtime открывает только назначенный дашборд без header, rail, Диспетчера событий, личных оповещений и остальных сервисов;
- PIN сотрудника включается политикой профиля, а ограниченный профиль может работать без персональной идентификации; offline всегда блокирует команды и не накапливает нажатия.

Параметр `?terminal={id}` используется только для локального preview и не является аутентификацией панели. В рабочем продукте device identity должна поступать из доверенной зарегистрированной сессии.

Все значения и события являются тестовыми. Команды не передаются реальному оборудованию.

Полное графическое редактирование, реальный IAM/RBAC, пользовательский CRUD, секреты интеграций, live-диагностика, сертификаты терминалов, настоящий heartbeat и audit backend в текущую итерацию не входят. Kiosk реализован как фиксированный логический сценарий, а не как общая адаптивная версия интерфейса.

Последний маршрут дашборда, избранное, разрешённый состав Рабочего стола, рабочий статус и редактируемые поля профиля сохраняются локально в браузере. Для проверки сценария первого посещения удалите данные сайта или соответствующие записи `localStorage`.
